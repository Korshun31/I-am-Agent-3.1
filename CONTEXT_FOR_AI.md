## 🏢 ПЛАН: Командная система (Premium) — v2

### Концепция
Агент с тарифом **Premium** может переключить аккаунт в режим **"Компания"** и пригласить до 3 агентов бесплатно для тестирования. В будущем — платно за каждого агента, чем больше пакет — тем дешевле за единицу. Агенты видят базу компании в **ограниченном режиме** — чтобы понять что есть для клиентов, не видя конфиденциальной информации.

---

### Роли

| Роль | Кто | Доступ |
|---|---|---|
| **Admin** | Владелец компании | Полный доступ ко всем объектам + управление командой |
| **Agent** | Сотрудник | Свои объекты — полный доступ. Чужие объекты компании — ограниченный просмотр |

### Что агент видит по чужим объектам компании

**Видит:** название, фото, тип, город/район (без точного адреса), удобства, кол-во спален, площадь, **цена аренды**, календарь (только занято/свободно — без имён и деталей)

**Не видит:** точный адрес, контакт собственника, комиссии, депозиты, детали бронирований (имя клиента, суммы, контакты)

### Этап 2 — Режим "Компания" в аккаунте

**Существующий UI (веб + мобильное) — усовершенствуем, не заменяем:**
- Переключатель "Частный агент / Компания" — оставляем
- Поля название, телефон, email компании — оставляем, они теперь создают запись в таблице `companies`
- Ниже блока компании появляется новый раздел **"Команда"** (только когда workAs = 'company')

**Блокировка переключения обратно в "Частный":**

| Ситуация | Действие |
|---|---|
| В команде 0 активных агентов | Переключение разрешено. Запись компании архивируется (status = inactive). При возврате в "Компания" — данные восстанавливаются |
| Есть приглашения со статусом "Отправлено/Ожидает" (не принятые) | Разрешено — приглашения аннулируются автоматически |
| Есть хотя бы 1 активный агент | **Заблокировано.** Показываем: "Нельзя перейти в режим частного агента пока в команде есть активные участники. Сначала удалите всех агентов." |

**Защита при удалении последнего агента:**
Показываем предупреждение: "После удаления вы сможете перевести аккаунт обратно в режим частного агента. Данные компании сохранятся. Продолжить?"

**Premium-защита:**
Если пользователь не Premium → при нажатии "Компания" показываем экран "Это Premium функция" вместо формы.

### Этап 3 — Система приглашений (двойная защита)

**Поток:**
1. Admin вводит email агента в разделе "Команда"
2. Система проверяет: **если email уже зарегистрирован → ошибка** "Этот email уже есть в системе. Попросите агента указать другой email" (личная база агента — его собственность, принудительное объединение запрещено)
3. Система генерирует: уникальный `invite_token` (в ссылке) + отдельный `secret_code` (Admin передаёт лично через WhatsApp/звонок)
4. На email агента уходит письмо: ссылки на App Store / Google Play + ссылка на веб версию
5. Агент переходит по ссылке → сразу всплывает окно **"Введите код доступа"**
6. Верный код → окно **"Создайте пароль"**: пароль / повтор пароля / подсказка к паролю (необязательно)
7. При переходе по ссылке: если у агента уже есть аккаунт — предупреждение **"Ваши объекты станут видны участникам команды в ограниченном режиме. Войти в команду?"** (Да / Нет)
8. Агент внутри команды ✅

**Статусы приглашения (видит Admin в разделе "Команда"):**
- `Отправлено` — письмо ушло, агент ещё не переходил по ссылке
- `Ожидает` — перешёл по ссылке, код не введён
- `Активен` — принял, работает в команде
- Кнопка **"Отозвать"** — доступна пока статус ≠ "Активен"

*Примечание: даже если письмо попало не туда — без секретного кода войти невозможно. Двойная защита.*

---

### Этап 4 — Управление командой (раздел "Команда" в аккаунте)

**Веб (полная версия):**
- Список участников: аватар, имя, роль, статус, кол-во объектов
- Кнопки: Пригласить, Отозвать приглашение, Удалить агента
- Секретный код: отображается Admin после создания приглашения (копировать + отправить)
- При удалении агента → его объекты переходят под ответственность "Компания"

**Мобильное (упрощённая версия):**
- Список участников с аватарами и статусами
- Те же базовые действия (Пригласить, Удалить)
- Секретный код — скопировать и отправить

---

### Этап 5 — Поле "Ответственный"

**Иерархия (каскад):**
- Резорт: задан ответственный → автоматически ответственный за все объекты внутри
- Объект внутри резорта: можно переопределить ответственного индивидуально
- Апартаменты: та же схема что резорты
- Отдельный дом: одно поле "Ответственный"

**Правила назначения:**
- Агент добавляет объект → он сам становится ответственным
- Admin добавляет объект → "Компания" по умолчанию (Admin может изменить)
- Агент уходит из команды → все его объекты переходят → "Компания"
- Admin вручную переназначает "Компания" → конкретному агенту

**UI:** выпадающий список участников команды в карточке объекта/резорта. Реализуется одновременно в вебе и мобильном.

---

### Этап 6 — Права доступа к объектам

| Кто | Какой объект | Доступ |
|---|---|---|
| Admin | Любой объект компании | Полный + редактирование + смена ответственного |
| Агент | Свой объект (ответственный = он) | Полный |
| Агент | Чужой объект компании | Ограниченный просмотр (см. "Что агент видит" выше) |

**В списке объектов:** фильтр **"Мои / Команда / Все"**
**Чужая карточка:** те же экраны что и обычно, но запрещённые блоки скрыты
**Чужой календарь:** показывает только занятые/свободные дни, без имён и деталей

---

### Этап 7 — Модерация объектов агента

1. Агент добавляет объект → статус `pending`, объект невидим для остальных
2. **Автопроверка дублей (V1):**
   - Нечёткое совпадение по названию
   - Совпадение по телефону контакта собственника
3. Admin получает push: *"Агент [Имя] добавил объект [Название]. Совпадений по названию: нет. Совпадений по контакту: нет. Рекомендую добавить."*
4. Admin видит очередь модерации (веб + мобильное)
5. **Одобрить →** статус `approved` + push агенту + опциональный push всей команде "Новый объект в базе"
6. **Отклонить →** Admin пишет причину → push агенту с причиной → агент исправляет и переотправляет

*(V2 — проверка по координатам 200м через Google Maps API)*

---

### Этап 8 — Уведомления

| Событие | Кому |
|---|---|
| Агент добавил объект на проверку | Admin (push с результатами автопроверки) |
| Объект одобрен | Агент (push) |
| Объект отклонён | Агент (push с причиной) |
| Новый объект одобрен | Вся команда (push, если включено в настройках) |
| Бронирование добавлено на "его" объект | Ответственный агент (push) |

### Порядок реализации

1. **Этап 1** — база данных (фундамент)
2. **Этапы 2 + 3** — режим компании + система приглашений
3. **Этап 4** — управление командой (UI)
4. **Этапы 5 + 6** — ответственный + права доступа
5. **Этапы 7 + 8** — модерация + уведомления

Каждый этап — **одновременно для веб и мобильного** чтобы избежать расхождений.

## Что это

Мобильное приложение для **агента по недвижимости** (органайзер/личный кабинет).  
Стек: **Expo SDK 54 + React Native**. Запуск: веб или Expo Go на телефоне.

## Навигация по экранам

1. **Preloader** — логотип (5 полос), «I am Agent», «Загрузка…» ~2.5 сек.
2. **Login** — E-mail, Password; Login, Sign up; соц. входы.
3. **Registration** — Name, E-mail, Password; Create account, Back.
4. **Main** — нижняя навигация (4 вкладки). По умолчанию **Account**. Остальные — заглушки.

## AccountScreen — личный кабинет

- **Шапка:** заголовок Account, кнопка выхода.
- **Аватар и имя** — при наличии.
- **My details** — заполненные поля (телефон, email, Telegram, WhatsApp и т.д.); кнопка редактирования (карандаш) → `MyDetailsEditModal`.
- **Settings (раскрываемый блок):**
  - Язык → `LanguageModal` (en, th, ru)
  - Уведомления → `NotificationsModal`
  - Валюта → `CurrencyModal`
- **Locations (раскрываемый блок):** список локаций, кнопка «+» → `AddLocationsModal` (country-state-city).
- **Contacts** — список контактов (телефон, email), клик — звонок/SMS/почта.

## Хранилище агентов

- **Файл:** `agents.json` в `documentDirectory` (expo-file-system).
- **Модуль:** `src/services/agentsStorage.js`
  - `saveAgent(agent)` — сохранить агента (с паролем).
  - `getAgentByEmail(email)` — получить по email.
  - `agentToUser(agent)` — отдать данные без пароля для UI.
- **Регистрация:** валидация email/пароля; проверка «email уже есть»; сохранение; переход в Main с `user`.
- **Логин:** поиск по email; при неверном пароле — алерт; иначе переход в Main с профилем.

## Состояние user в App.js

```js
user = {
  email, name, lastName, phone, telegram,
  documentNumber, extraPhones, extraEmails,
  whatsapp, photoUri
}
```

- Передаётся в `MainScreen` → `AccountScreen`.
- Обновление: `onUserUpdate(updatedUser)` → `handleUserUpdate` в App.js.

## Контекст языка

- `src/context/LanguageContext.js` — `language`, `setLanguage`, `t(key)`.
- Переводы: `src/i18n/translations.js` (en, th, ru).
- При смене языка — сохранение в AsyncStorage (fallback при ошибке).
