# Module: Contacts

## Скоуп

Модуль покрывает: создание, редактирование и удаление контактов (клиенты и собственники), видимость контактов для разных ролей, фото
 и документы контактов.

**НЕ покрывается:**
- Бронирования (→ Bookings, но связь через `contact_id` описана здесь)
- Объекты (→ Properties, но связь через `owner_id` описана здесь)

**Файлы модуля:**
- `src/services/contactsService.js` — CRUD, фото, миграция фото
- `src/components/AddContactModal.js` — создание/редактирование (мобильный)
- `src/screens/ContactsScreen.js` — список контактов (мобильный)
- `src/screens/ContactDetailScreen.js` — детали контакта (мобильный)
- `src/web/components/WebContactEditPanel.js` — создание/редактирование (веб)
- `src/web/screens/WebContactsScreen.js` — список + детали (веб)

**DB:**
- FK `contacts_company_id_fkey` → `companies(id)` ON DELETE SET NULL
- FK `contacts_user_id_fkey` → `users_profile(id)` ON DELETE CASCADE
- Нет триггеров на таблице contacts
- Нет CHECK constraint на `type` (TD-098)
- Фото в отдельном бакете `contact-photos`

**Таблицы ссылающиеся на contacts:**
- `properties.owner_id` → ON DELETE SET NULL
- `properties.owner_id_2` → ON DELETE SET NULL
- `bookings.contact_id` → ON DELETE SET NULL

## Подмодуль 1: Типы контактов и создание (CT-CR)

### Два типа

**CT-CR-1.** Два типа контактов:
- **clients** (клиенты/арендаторы) — гости которые снимают жильё
- **owners** (собственники) — владельцы недвижимости

**CT-CR-1.1.** *(TD-098)* Нет CHECK constraint на `contacts.type`. Добавить: `CHECK (type IN ('clients', 'owners'))`.

**CT-CR-2.** Тип задаётся при создании и **не может быть изменён** при редактировании. Заблокирован на обеих платформах (`lockType`
на вебе).

### Обязательные поля

**CT-CR-3.** Обязательное поле: **Имя**. Проверяется на клиенте (обе платформы).

**CT-CR-3.1.** *(TD-101, закрыт 2026-04-27, миграция `20260427000006_contacts_name_check.sql`)* В БД добавлен `CHECK (trim(name) <> '')` (constraint `contacts_name_not_blank`). Прямой REST-вызов с пустым/пробельным именем отклоняется PostgreSQL ошибкой 23514. JS-валидация в формах сохранена как первая линия.

### Поля контакта

**CT-CR-4.** Все поля кроме имени опциональные:
- Фамилия, телефон, email, Telegram (массив), WhatsApp (массив), номер документа (паспорт), национальность, дата рождения, фото,
доп. телефоны (массив), доп. email (массив), документы (массив файлов)

### Автоматические поля

**CT-CR-5.** При создании:
- `user_id` = создатель
- `company_id` = из `resolveCompanyId()` (компания создателя)
- Без `company_id` — ошибка `CONTACT_NO_COMPANY`

### Permissions

**CT-CR-6.** Отдельного permission для работы с контактами **нет**. Агент всегда может создавать/редактировать/удалять свои
контакты.

**CT-CR-6.1.** *(TD-102 — ✅ СНЯТ 2026-04-30, этап 2)* `can_manage_clients` удалён из CURSOR_RULES 4.2 и из всех проверок в коде. Ключ физически остаётся в JSONB до этапа 3 (cleanup-миграция). Агент всегда работает со своими контактами без отдельной галочки.

### Фото контакта

**CT-CR-7.** Фото хранятся в бакете `contact-photos`. Загрузка через `uploadContactPhoto()`.

**CT-CR-7.1.** *(TD-100)* Нет сжатия фото при загрузке. Добавить: 1200px + миниатюра 150px, JPEG 0.85 (как для объектов). Обе
платформы.

**CT-CR-7.2.** *(TD-104)* На вебе нет поля "Фото контакта" (на мобильном есть). Добавить.

### Документы

**CT-CR-8.** Можно прикрепить файлы к контакту (скан паспорта, договор и т.д.).

**CT-CR-8.1.** *(TD-103)* На мобильном нет возможности прикрепить документы (на вебе есть). Добавить.

### Механика ввода контактных данных

**CT-CR-9.** Разная механика на платформах — допустимая разница:
- Мобильный: отдельные поля для каждого типа (телефон, email, telegram, whatsapp) + кнопка "добавить ещё"
- Веб: универсальный список "Методы связи" — пользователь добавляет строки и выбирает тип

Данные одинаковые в БД.

## Подмодуль 2: Редактирование и удаление (CT-ED)

### Кто может редактировать

**CT-ED-1.** Админ — редактирует любой контакт компании.

**CT-ED-2.** Агент — редактирует **только свои** контакты (`user_id = auth.uid()`). RLS: `own update in company`.

**CT-ED-3.** Тип контакта заблокирован при редактировании (CT-CR-2).

### Кто может удалить

**CT-ED-4.** Админ — удаляет любой контакт компании.

**CT-ED-5.** Агент — удаляет **только свои** контакты (`user_id = auth.uid()`). RLS: `own delete in company`.

### Предупреждения при удалении

**CT-ED-6.** *(TD-105)* При удалении контакта-собственника: если к нему привязаны объекты — предупреждение: "У этого контакта N
объектов. При удалении они останутся без собственника." Обе платформы. Сейчас не реализовано.

**CT-ED-7.** *(TD-106)* При удалении контакта-клиента: если к нему привязаны бронирования — предупреждение: "У этого контакта N
бронирований. При удалении контакт будет отвязан от бронирований." Обе платформы. Сейчас не реализовано.

### Каскадные последствия при удалении

**CT-ED-8.** При удалении контакта:
- `properties.owner_id → NULL` (объекты остаются без собственника)
- `properties.owner_id_2 → NULL` (то же для второго собственника)
- `bookings.contact_id → NULL` (бронирования остаются без контакта)

Объекты и бронирования **не удаляются**.

## Подмодуль 3: Видимость контактов (CT-VIS)

### Правила видимости

**CT-VIS-1.** Админ видит **все** контакты компании. RLS: `owner full read` (`auth_is_company_owner`).

**CT-VIS-2.** Агент видит:
1. Контакты которые **сам создал** (`user_id = auth.uid()`)
2. **Собственников** объектов за которые отвечает (RLS: `agent reads property owners`)
3. **Клиентов** из бронирований за которые отвечает *(TD-099 — не реализовано в RLS)*

**CT-VIS-2.1.** *(TD-099)* Пункт 3 не реализован в RLS. Нужна политика: агент видит контакт если `contacts.id` в
`bookings.contact_id` где `bookings.booking_agent_id = auth.uid()`. Реализовать после TD-086.

**CT-VIS-2.2.** *(TD-107)* На вебе агент загружает клиентов из бронирований через JS-логику (обходя RLS). После реализации TD-099 —
убрать JS-костыль, использовать единый `getContacts()` на обеих платформах.

### RLS-политики

**CT-VIS-3.** Текущие политики:

| Политика | Кто | Что |
|---|---|---|
| `owner full read` | Админ | Читает все контакты компании |
| `agent read own` | Агент | Читает свои (`user_id = uid` + member компании) |
| `agent reads property owners` | Агент | Читает собственников своих объектов |
| `own insert in company` | Все | Создаёт |
| `own update in company` | Создатель | Обновляет свои |
| `own delete in company` | Создатель | Удаляет свои |

### Сравнение веб vs мобильный

| Аспект | Мобильный | Веб | Совпадает? |
|---|---|---|---|
| Админ видит всё | ✅ | ✅ | ✅ |
| Агент видит свои | ✅ через RLS | ✅ через `getMyContacts()` | ✅ |
| Агент видит собственников | ✅ через RLS | ✅ через JS (getContactsByIds) | ✅ Разный подход |
| Агент видит клиентов бронирований | ❌ Ограничено RLS | ✅ Через JS-костыль (TD-107) | ⚠️  |
