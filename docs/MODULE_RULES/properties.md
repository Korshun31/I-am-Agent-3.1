# Module: Properties (Real Estate)

> Документ в процессе написания. Покрыты подмодули 1-2 из 9.

## Скоуп

Модуль покрывает: типы объектов и иерархию, создание, редактирование, модерацию (approve/reject), удаление, фотографии, локации и районы, назначение ответственного агента, видимость объектов.

**НЕ покрывается:**
- Бронирования объектов (→ Bookings)
- Контакты собственников (→ Contacts, но связь через `owner_id` описана здесь)
- Управление командой (→ Company & Team, но пересечение через `responsible_agent_id` и permissions)

**Файлы модуля:**
- `src/services/propertiesService.js` — CRUD, черновики, модерация, каскадные обновления
- `src/services/storageService.js` — загрузка фотографий в Supabase Storage
- `src/services/locationsService.js` — локации и районы
- `src/screens/PropertyDetailScreen.js` — детальный экран объекта (мобильный, 2672 строки)
- `src/screens/RealEstateScreen.js` — список объектов (мобильный)
- `src/components/PropertyEditWizard.js` — пошаговый визард создания/редактирования (мобильный)
- `src/web/screens/WebPropertiesScreen.js` — список + детали (веб, 2481 строка)
- `src/web/components/WebPropertyEditPanel.js` — боковая панель создания/редактирования (веб)

**DB-триггеры:**
- `trg_auto_set_property_company` → `auto_set_property_company()` — на INSERT, автоматически подставляет `company_id` ❌ НЕ в миграциях (TD-025). Содержит мёртвый код со статусом 'submitted' (TD-052)

**DB constraints:**
- `properties_name_not_blank` — CHECK: name не пустое
- `properties_property_status_check` — CHECK: status IN ('pending', 'approved', 'rejected')
- `properties_type_check` — CHECK: type IN ('house', 'resort', 'condo', 'resort_house', 'condo_apartment')
- `properties_resort_id_fkey` — FK: resort_id → properties(id) ON DELETE CASCADE

## Подмодуль 1: Типы объектов и иерархия (PR-TYPE)

### Типы

Два уровня объектов:

**Самостоятельные (верхний уровень):**
- **House** — отдельный дом, не входит в комплекс
- **Resort** — комплекс, контейнер для домов
- **Condo** — здание, контейнер для квартир

**Дочерние (вложены в контейнер):**
- **Resort House** — дом внутри резорта (`resort_id` = ID резорта)
- **Condo Apartment** — квартира внутри кондо (`resort_id` = ID кондо)

Все объекты хранятся в одной таблице `properties`. Различаются полем `type` и наличием `resort_id`.

### Правила

**PR-TYPE-1.** Тип объекта задаётся при создании и **не может быть изменён**. Дом не может стать резортом, квартира не может стать домом.

**PR-TYPE-2.** Дочерний объект **не может существовать** без родителя. При удалении родителя (резорт/кондо) каскадно удаляются все его дочерние объекты. Обеспечивается FK constraint `properties_resort_id_fkey` с `ON DELETE CASCADE`. UI показывает двойное подтверждение с предупреждением.

**PR-TYPE-3.** Ограничений на количество дочерних объектов в резорте/кондо нет.

**PR-TYPE-4.** *(TD-043, низкий)* Поле `resort_id` используется для связи и с резортами, и с кондо — название не отражает суть. Рекомендуется переименовать в `parent_id`. Масштаб: 78 вхождений в 16 JS-файлах + FK + триггер. Делать только при наличии тестов.

### Наследование дочерних от родителя

| Поле | Наследуется? | Каскадное обновление? |
|---|---|---|
| Район (district) | Да, read-only | Да, `updateResortChildrenDistrict()` |
| Ответственный агент | Да, read-only | Да, `updatePropertyResponsible(cascade=true)` |
| Локация (location_id) | Да, при создании | Нет явного каскада |
| Собственник (owner_id) | Да, при создании | Нет |

### Сортировка дочерних

Дома в резорте / квартиры в кондо сортируются по `code_suffix`: числовые по возрастанию (30, 35, 37), буквенные по алфавиту. Функция `sortByInternalCode()`.

## Подмодуль 2: Создание объекта (PR-CR)

### Обязательные поля

**PR-CR-1.** Обязательные поля при создании: **Название**, **Локация (город)**, **Район**. Все остальные опциональные — дозаполняются позже.

**PR-CR-1.1.** *(TD-044)* На мобильном сейчас обязательно только Название. Нужно добавить проверку Локации и Района (как на вебе).

**PR-CR-1.2.** *(TD-050)* Обязательность Локации проверяется только на клиенте (JS). В БД нет CHECK constraint на `location_id IS NOT NULL`. Нужно добавить серверную валидацию.

### Тип и статус

**PR-CR-2.** Тип объекта задаётся при создании и не меняется (см. PR-TYPE-1).

**PR-CR-3.** Статус при создании:
- Админ → `'approved'` (сразу виден, модерация не нужна)
- Агент → `'pending'` (требует одобрения админа)

**PR-CR-3.1.** Создание объекта агентом возможно **только** при наличии permission `can_add_property` (см. CO-PERM-1, CURSOR_RULES раздел 4.1). *(TD-005: permissions не полностью enforced в UI)*

### Автоматические поля

**PR-CR-4.** `company_id` подставляется автоматически:
- Для агента — из `company_members` через `resolveAgentCompanyId()`
- Для админа — триггером `auto_set_property_company` (если JS-код не подставил)

**PR-CR-5.** `responsible_agent_id` при создании = сам создатель.

**PR-CR-6.** `user_id` при создании = сам создатель. Это фиксация "кто создал", не право доступа (см. CURSOR_RULES CF-001).

### Уникальность кода

**PR-CR-7.** Код объекта (`code`) должен быть **уникальным в рамках компании** для объектов верхнего уровня (house, resort, condo). У разных компаний могут быть одинаковые коды.

**PR-CR-7.1.** Внутренний код (`code_suffix`) дочернего объекта должен быть **уникальным в рамках одного родителя**. Нельзя иметь два дома с одинаковым номером в одном резорте.

**PR-CR-7.2.** *(TD-054)* Уникальность кода сейчас не проверяется ни на клиенте, ни в БД. Нужно добавить: UNIQUE constraint в БД + проверку в UI перед сохранением.

### Визард мобильный — шаги по типам

**PR-CR-8.** Для типов house / resort_house / condo_apartment — **8 шагов**:

| Шаг | Название | Содержимое |
|---|---|---|
| 1 | Info | Название, город/локация, район, код (или code_suffix для дочерних), Google Maps, адрес, собственник (+ доп. собственник для дочерних), ответственный агент (только для админа) |
| 2 | Chars | Спальни, ванные, площадь, этаж (только condo_apartment), расстояние до пляжа/магазина |
| 3 | Desc | Описание |
| 4 | Media | Фотографии (лимит по тарифу: Standard=10, Premium/Korshun=30), видео (URL-ссылки) |
| 5 | Amenities | Удобства (переключатели): бассейн, зал, парковка, интернет, ТВ, стиральная, посудомойка, холодильник |
| 6 | Additional | Кондиционеры, скорость интернета, животные (переключатель), долгосрочная аренда (переключатель) |
| 7 | Pricing | Аренда, депозиты, комиссии, комиссия собственнику (разовая/ежемесячная, сумма или %), коммуналка (электричество, вода с типом, газ, интернет, клининг, выездной клининг) |
| 8 | Comments | Ссылка на сайт объекта, комментарии |

**PR-CR-9.** Для типов resort / condo — **5 шагов**: Info, Chars, Desc, Media, Comments. Без Amenities, Additional, Pricing — контейнеры не сдаются в аренду.

**PR-CR-10.** Отличие Chars для контейнеров:
- Resort: поле "Количество домов" (`houses_count`) вместо спален/ванных/площади
- Condo: поле "Количество этажей" (`floors`) вместо спален/ванных/площади

Для контейнеров (resort/condo) — только Main и Photos.

### Дочерние объекты — особенности создания

**PR-CR-12.** Дочерние объекты (resort_house, condo_apartment) при создании:
- `resort_id` = ID родителя
- Город, район, собственник, ответственный агент — наследуются от родителя (отображаются read-only)
- Используется `code_suffix` вместо `code`
- Доступен дополнительный собственник (`owner_id_2`)
- Расстояние до пляжа/магазина — наследуется от родителя если не задано
- Condo apartment — дополнительное поле "Этаж" (`floor_number`)

### Видимость полей по ролям

**PR-CR-13.** Видимость полей при создании:

| Поле | Админ | Агент |
|---|---|---|
| Собственник | ✅ видно | ✅ видно |
| Ответственный агент | ✅ видно | ❌ скрыто |

Одинаково на вебе и в мобильном приложении.

### Расхождения веб vs мобильный (к исправлению)

| # | Расхождение | TD |

| 1 | Обязательные поля: мобильный не проверяет локацию и район | TD-044 |
| 2 | Видео: веб — одна ссылка (`video_url`), мобильный — массив (`videos`) | TD-045 |

### Безопасность

**PR-CR-14.** *(TD-050)* Обязательные поля проверяются только на клиенте. В БД нет constraint на `location_id`. Прямой запрос к API может создать объект без локации.

**PR-CR-15.** *(TD-051)* `createPropertyFull()` принимает произвольные поля через spread (`...updates`). Нет фильтрации допустимых полей перед INSERT. Рекомендация: whitelist разрешённых полей.

**PR-CR-16.** *(TD-052)* Триггер `auto_set_property_company` содержит мёртвый код: пытается установить статус `'submitted'`, которого нет в CHECK constraint. JS-код всегда ставит статус раньше триггера. Мёртвая логика не выполняется, но должна быть удалена.

**PR-CR-17.** *(TD-053)* При удалении объекта фотографии остаются в Supabase Storage. `deleteProperty()` не вызывает `deletePhotoFromStorage()`. Утечка места.

**PR-CR-18.** Лимит фотографий (Standard=10, Premium=30) проверяется на клиенте. Серверная проверка отсутствует. Достаточно проверки в `propertiesService.js` перед сохранением.

## Подмодуль 3: Редактирование объекта (PR-EDIT)

### Админ

**PR-EDIT-1.** Админ может редактировать **все поля** объекта, кроме типа (PR-TYPE-1). Включая код, локацию, район, собственника,
ответственного, цены, фото — без ограничений.

**PR-EDIT-2.** Изменения админа **применяются сразу** — без модерации, без черновиков. Статус объекта не меняется.

### Агент — разрешения

**PR-EDIT-3.** Два **независимых** разрешения контролируют редактирование агентом:

| Разрешение | Что контролирует |
|---|---|
| `can_edit_info` | Информация: название, описание, фото, удобства, характеристики, локация, адрес |
| `can_edit_prices` | Цены: аренда, депозиты, комиссии, коммуналка |

**PR-EDIT-4.** Комбинации разрешений:

| `can_edit_info` | `can_edit_prices` | Поведение |
|---|---|---|
| ON | ON | Агент редактирует всё, сохраняет напрямую без модерации |
| ON | OFF | Информацию сохраняет напрямую, изменения цен идут через черновик на проверку админу |
| OFF | ON | Цены сохраняет напрямую, изменения информации идут через черновик на проверку админу |
| OFF | OFF | Кнопка "Редактировать" **не видна**. Агент не может редактировать |

**PR-EDIT-5.** Объект со статусом `rejected` — все изменения идут через черновик, независимо от разрешений. Потому что rejected
требует повторного одобрения админом.

### Агент — видимость полей

**PR-EDIT-6.** Агент **видит все поля** при редактировании, **кроме поля "Ответственный агент"** — оно скрыто для агента (одинаково
при создании PR-CR-13 и при редактировании). Остальные поля видны независимо от разрешений. Агент может менять любые видимые поля.
При сохранении система разделяет:
- Поля на которые **есть** разрешение → сохраняются напрямую
- Поля на которые **нет** разрешения → уходят в черновик на проверку админу

**PR-EDIT-7.** Если агент поменял и информацию, и цены — а разрешение есть только на одно:
- Разрешённые изменения → сохраняются сразу
- Неразрешённые изменения → черновик → админу уведомление

### Система черновиков (drafts)

**PR-EDIT-8.** Черновик хранится в таблице `property_drafts`:
- `property_id` — объект
- `user_id` — агент
- `draft_data` — JSONB с изменёнными полями
- `status` — `pending` / `approved` / `rejected`
- `rejection_reason` — причина отклонения (если rejected)
- Один черновик на пару property_id + user_id (upsert)

**PR-EDIT-9.** При отправке черновика:
1. Создаётся/обновляется запись в `property_drafts` (status = `pending`)
2. Статус объекта меняется на `pending` (чтобы админ видел "на проверке")
3. Админу отправляется уведомление "Агент отправил изменения для объекта X"
4. `broadcastChange('properties')` — обновление у всех участников

**PR-EDIT-10.** При одобрении черновика админом:
1. Данные из `draft_data` применяются к объекту (`UPDATE properties SET ...`)
2. Статус объекта → `approved`
3. Статус черновика → `approved`

**PR-EDIT-11.** При отклонении черновика админом:
1. Статус объекта → `rejected`
2. Статус черновика → `rejected` с причиной
3. Агенту отправляется уведомление с причиной отклонения
4. Запись в журнал отклонений (`property_rejection_history`)

### Просмотр черновика админом

**PR-EDIT-12.** *(TD-055)* При просмотре черновика админ должен видеть **сравнение "было / стало"**:
- Названия изменённых полей выделены красным цветом
- Для каждого изменённого поля: старое значение сверху, новое снизу
- Неизменённые поля не выделяются
Сейчас не реализовано — админ видит черновик, но без визуального сравнения.

### Каскадные обновления при редактировании

**PR-EDIT-13.** При изменении `responsible_agent_id` на резорте/кондо — каскадно обновляется на всех дочерних объектах.

**PR-EDIT-14.** При изменении `district` на резорте/кондо — каскадно обновляется район всех дочерних объектов.

### Расхождения веб vs мобильный

**PR-EDIT-15.** *(TD-056)* Логика `needsApproval` различается:
- Мобильный: проверяет `!canEditInfo OR !canEditPrices`
- Веб: проверяет только `!canEditInfo`
Нужно привести к единой логике согласно PR-EDIT-4.

**PR-EDIT-16.** *(TD-057)* Сейчас в UI один переключатель "Редактирование объектов" вместо двух ("Редактирование информации" и
"Редактирование цен"). Нужно разделить в `WebTeamSection` и мобильном аналоге.

**PR-EDIT-17.** *(TD-058)* Раздельное сохранение (PR-EDIT-6, PR-EDIT-7) — разрешённые поля напрямую, неразрешённые через черновик —
сейчас не реализовано. Работает "всё или ничего": либо всё напрямую, либо всё через черновик.

## Подмодуль 4: Модерация (PR-MOD)

### Два типа модерации

**PR-MOD-1.** Модерация нового объекта (`property_submitted`):
1. Агент создаёт объект → статус `pending`
2. Админу уведомление типа `property_submitted`
3. Админ открывает объект → видит статус "На проверке" → кнопки "Одобрить" / "Отклонить"
4. **Одобрить** → `approveProperty(id)` → статус `approved` → агенту уведомление `property_approved`
5. **Отклонить** → `rejectProperty(id, reason)` → статус `rejected` + причина → агенту уведомление `property_rejected` → запись в
`property_rejection_history`

**PR-MOD-2.** Модерация черновика изменений (`edit_submitted`):
1. Агент редактирует объект → черновик в `property_drafts` → статус объекта `pending`
2. Админу уведомление типа `edit_submitted`
3. Админ открывает объект → видит что есть черновик → кнопки "Одобрить" / "Отклонить"
4. **Одобрить** → `approvePropertyDraft(draftId)` → данные из черновика применяются к объекту → статус `approved` → статус черновика
 `approved` → агенту уведомление `edit_approved`
5. **Отклонить** → `rejectPropertyDraft(draftId, reason)` → статус объекта `rejected` → статус черновика `rejected` + причина →
агенту уведомление `edit_rejected` → запись в `property_rejection_history`

### Авто-одобрение при редактировании админом

**PR-MOD-3.** Когда админ редактирует объект со статусом `rejected` — статус автоматически меняется на `approved`. Агенту
отправляется уведомление `property_approved`. Это логично: если админ сам исправил объект, повторная модерация не нужна.

**PR-MOD-3.1.** *(TD-059)* На мобильном авто-одобрение при редактировании rejected объекта **не реализовано** — статус остаётся
`rejected` после сохранения. На вебе работает правильно. Нужно добавить на мобильном.

### Как админ находит объекты на проверке

**PR-MOD-4.** Три способа:
1. **Значок ⏳ в списке объектов** — визуально выделяет pending/rejected объекты
2. **Колокольчик уведомлений** — push-уведомления от агентов (`PropertyNotificationsModal` / `WebNotificationBell`)
3. **Фильтр "На проверке"** — показывает только объекты со статусом pending/rejected

**PR-MOD-4.1.** Фильтр реализован по-разному:
- Мобильный: чекбокс `inReview` в модалке фильтра (воронка)
- Веб: отдельный таб "На проверке (N)" с количеством объектов
Разная механика, одинаковый результат — допустимая разница в UX.

### Отклонение — обязательная причина

**PR-MOD-5.** При отклонении админ **обязан** указать причину (пустая строка не принимается). Причина:
- Сохраняется в `properties.rejection_reason`
- Сохраняется в `property_rejection_history` (журнал, append-only)
- Отправляется агенту в уведомлении

### Журнал отклонений

**PR-MOD-6.** `property_rejection_history` — append-only таблица:
- `property_id` — объект
- `reason` — причина (NOT NULL)
- `rejection_type` — `'property_submitted'` или `'edit_submitted'`
- `rejected_by` — ID админа
- `created_at` — дата

Агент и админ могут видеть полную историю отклонений на экране объекта.

### Типы уведомлений

**PR-MOD-7.** Уведомления при модерации:

| Тип | Когда | Кому |
|---|---|---|
| `property_submitted` | Агент создал новый объект | Админу |
| `edit_submitted` | Агент отправил черновик изменений | Админу |
| `property_approved` | Админ одобрил новый объект | Агенту |
| `edit_approved` | Админ одобрил черновик | Агенту |
| `property_rejected` | Админ отклонил новый объект | Агенту |
| `edit_rejected` | Админ отклонил черновик | Агенту |

**PR-MOD-7.1.** В коде также упоминаются типы `price_submitted` / `price_approved` / `price_rejected`. Сейчас не используются —
зарезервированы для раздельного сохранения цен (TD-058). Когда TD-058 будет реализован, появится отдельный тип уведомления для
ценовых изменений.

### Просмотр черновика на вебе — режим reviewMode

**PR-MOD-8.** На вебе админ может открыть `WebPropertyEditPanel` в режиме `reviewMode`:
- Все поля `readOnly` — нельзя редактировать
- Внизу кнопки "Одобрить" / "Отклонить" (вместо "Сохранить")
- При отклонении — инлайн-поле для ввода причины

На мобильном аналога `reviewMode` нет — админ одобряет/отклоняет из экрана деталей объекта. Это допустимая разница: на мобильном
экран деталей достаточно информативен.

### RLS-политики

**PR-MOD-9.** `property_drafts`:
- Агент: INSERT/SELECT/UPDATE только своих (`user_id = auth.uid()`)
- Админ: SELECT/UPDATE черновиков объектов своей компании (`auth_is_company_owner`)

**PR-MOD-10.** `property_rejection_history`:
- Агент: SELECT — свои объекты + объекты где ответственный (`user_id = uid OR responsible_agent_id = uid`)
- Админ: SELECT + INSERT — объекты компании (`companies.owner_id = uid`)

### Расхождения веб vs мобильный

**PR-MOD-11.** *(TD-059)* Авто-одобрение rejected при редактировании админом — только на вебе. На мобильном статус остаётся
`rejected` после сохранения.

**PR-MOD-12.** *(TD-060)* Каскадное обновление района дочерних объектов при изменении района резорта/кондо — только на мобильном. На
 вебе `updateResortChildrenDistrict` не вызывается в `WebPropertyEditPanel`.

## Подмодуль 5: Удаление объекта (PR-DEL)

### Кто может удалить

**PR-DEL-1.** **Админ** может удалить любой объект — одобренный, на проверке, отклонённый. Без ограничений.

**PR-DEL-2.** **Агент** может удалить объект **только** при выполнении **обоих** условий одновременно (LOCK-001):
1. Он сам создал этот объект (`user_id = auth.uid()`)
2. Объект **не** одобрен (`property_status != 'approved'`)

Если объект одобрен — агент не может его удалить, даже если сам создавал. Одобренный объект — часть рабочей базы компании, удаление
только через админа.

Если объект создан другим агентом — удалить нельзя, даже если он pending.

### Подтверждение удаления

**PR-DEL-3.** Для обычного объекта — одно подтверждение: "Удалить объект?"

**PR-DEL-4.** Для контейнера (резорт/кондо) с одобренными дочерними объектами — **двойное** подтверждение:
1. "Удалить объект?"
2. "Внимание! Все дома/квартиры внутри тоже будут удалены"

### Каскадные последствия

**PR-DEL-5.** При удалении объекта каскадно удаляются (FK ON DELETE CASCADE в БД):
- Дочерние объекты (resort_house / condo_apartment)
- Бронирования всех удалённых объектов
- Уведомления по всем удалённым объектам
- Черновики изменений (`property_drafts`)
- История отклонений (`property_rejection_history`)

**PR-DEL-6.** Контакты собственника **не удаляются**. FK `properties.owner_id` → `contacts.id` настроен как `ON DELETE SET NULL`.
При удалении объекта контакт остаётся, просто теряет привязку к этому объекту.

### Предупреждения о бронированиях

**PR-DEL-7.** *(TD-061)* Если у объекта есть бронирования — при удалении должно показываться дополнительное предупреждение: "У этого
 объекта N бронирований. Они тоже будут удалены." Сейчас не реализовано — бронирования удаляются каскадно без предупреждения.

### Что происходит в коде при удалении

**PR-DEL-8.** `deleteProperty()` выполняет:
1. Defense-in-depth: если агент пытается удалить approved → ошибка (дублирует UI-проверку)
2. Удаляет незакрытые уведомления модерации (`property_submitted`, `edit_submitted`, `price_submitted` с `action_taken = false`) —
**избыточно**, FK CASCADE сделал бы то же
3. DELETE из `properties` → каскадное удаление всех связей
4. `broadcastChange('properties')`

### Очистка фотографий

**PR-DEL-9.** *(TD-053, обновлено)* При удалении объекта фотографии остаются в Supabase Storage. `deleteProperty()` не вызывает
`deletePhotoFromStorage()`. При реализации нужно учитывать каскад: перед удалением контейнера (резорт/кондо) — собрать фото **всех
дочерних объектов** и удалить из Storage, потом удалить фото самого контейнера.

### Сравнение веб vs мобильный

Логика удаления **совпадает** на обеих платформах:
- Одинаковые правила видимости кнопки (LOCK-001)
- Одинаковое двойное подтверждение для контейнеров
- Разница только в UI: мобильный — Alert, веб — модальные окна

## Подмодуль 6: Фотографии (PR-PHOTO)

### Хранение

**PR-PHOTO-1.** Фотографии хранятся в Supabase Storage, бакет `property-photos`. Путь: `{user_id}/{timestamp}_{random}.{ext}`. URL
фотографий сохраняются в массиве `properties.photos` (JSONB).

### Размеры и сжатие

**PR-PHOTO-2.** При загрузке фотографии создаются **два размера**:

| Размер | Назначение | Вес ~1 фото |
|---|---|---|
| **1200px** (по большей стороне) | Полноэкранный просмотр | 150-300 KB |
| **150px** (по большей стороне) | Миниатюры в списках и карточках | 10-20 KB |

Формат: JPEG, качество 0.85. Оба размера сохраняются в Storage.

**PR-PHOTO-2.1.** *(TD-062)* На вебе сейчас нет сжатия — фото загружаются как есть. Нужно добавить сжатие до 1200px.

**PR-PHOTO-2.2.** *(TD-064)* Миниатюры сейчас не генерируются ни на одной платформе. Везде загружается полноразмерное фото даже для
карточек 64-80px.

### Где какой размер используется

**PR-PHOTO-3.** Миниатюра (150px):
- Список объектов — карточка (~64-80px)
- Детали объекта — горизонтальная лента превью (~100x80px)
- Дочерний объект — превью в списке дочерних
- Панель/визард редактирования — превью фото
- Бронирования — фото в карточке бронирования

**PR-PHOTO-4.** Полное фото (1200px):
- Полноэкранная галерея (просмотр по свайпу)
- Детали объекта на вебе — крупная галерея (360x240px)

### Лимиты

**PR-PHOTO-5.** Максимум фотографий зависит от тарифа:

| Тариф | Максимум |
|---|---|
| Standard | 10 |
| Premium / Korshun | 30 |

Определяется через `getPhotoLimitForProperty(role)` в `roleFeatures.js`. Проверяется на клиенте. Серверная проверка отсутствует
(PR-CR-18).

### Загрузка

**PR-PHOTO-6.** Мобильный:
- Галерея устройства (`ImagePicker`), множественный выбор
- Сжатие через `ImageManipulator` (1200px, JPEG 0.85)
- Локальные URI сохраняются в форме → при сохранении объекта загружаются в Storage → получают публичные URL

**PR-PHOTO-7.** Веб:
- Стандартный файловый диалог (`input type=file, accept=image/*, multiple`)
- Файлы загружаются напрямую в Storage
- *(TD-062)* Нет сжатия — нужно добавить

### Удаление фото при редактировании

**PR-PHOTO-8.** При удалении фото из карточки объекта — фото **должно** удаляться из Storage на всех платформах.

- Мобильный: ✅ `handleWizardSave` сравнивает старые и новые фото → удалённые чистятся через `deletePhotoFromStorage()`
- *(TD-063)* Веб: ❌ `handleRemovePhoto` убирает из массива формы, но **не удаляет из Storage**. Утечка места.

### Полноэкранная галерея

**PR-PHOTO-9.** На обеих платформах должна быть полноэкранная галерея:
- Листание фото (свайп на мобильном, стрелки на вебе)
- Кнопка сохранения фото на устройство/компьютер
- Кнопка удаления фото

- Мобильный: ✅ `PhotoGalleryModal` — свайп, сохранение, удаление
- *(TD-065)* Веб: ❌ нет полноэкранной галереи со свайпом
- *(TD-066)* Веб: ❌ нет кнопок сохранения и удаления при просмотре фото

### Удаление фото при удалении объекта

**PR-PHOTO-10.** *(TD-053)* При удалении объекта фотографии должны удаляться из Storage. При удалении контейнера — включая фото всех
 дочерних объектов. Сейчас не реализовано.

### Видео

**PR-PHOTO-11.** Видео хранятся как URL-ссылки (YouTube и т.д.):
- Мобильный: массив `videos` — несколько ссылок
- *(TD-045)* Веб: одна ссылка `video_url` — нужно заменить на массив

### Расхождения веб vs мобильный

| # | Расхождение | TD |
|---|---|---|
| 1 | Веб: нет сжатия фото при загрузке | TD-062 |
| 2 | Веб: удалённые фото не чистятся из Storage | TD-063 |
| 3 | Нет миниатюр (обе платформы) | TD-064 |
| 4 | Веб: нет полноэкранной галереи со свайпом | TD-065 |
| 5 | Веб: нет кнопок сохранения/удаления при просмотре фото | TD-066 |

## Связь с RBAC

- `can_add_property` — агент может создать объект только при наличии этого permission (CURSOR_RULES 4.1)
- `can_edit_info`, `can_edit_prices` — два независимых разрешения для редактирования (PR-EDIT-3, PR-EDIT-4)
- `responsible_agent_id` — определяет видимость объекта для агента (подмодуль 9)
- LOCK-001 — правила удаления (подмодуль 5)
