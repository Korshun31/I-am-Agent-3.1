# Module: Properties (Real Estate)

> Документ в процессе написания. Покрыты подмодули 1-2 из 9.

## Скоуп

Модуль покрывает: типы объектов и иерархию, создание, редактирование, модерацию (approve/reject), удаление, фотографии, локации и районы, назначение ответственного агента, видимость объектов.

**НЕ покрывается:**
- Бронирования объектов (→ Bookings)
- Контакты собственников (→ Contacts, но связь через `owner_id` описана здесь)
- Управление командой (→ Company & Team, но пересечение через `responsible_agent_id` и permissions)

**Файлы модуля:**
- `src/services/propertiesService.js` — CRUD, черновики, модерация, каскадные обновления
- `src/services/storageService.js` — загрузка фотографий в Supabase Storage
- `src/services/locationsService.js` — локации и районы
- `src/screens/PropertyDetailScreen.js` — детальный экран объекта (мобильный, 2672 строки)
- `src/screens/RealEstateScreen.js` — список объектов (мобильный)
- `src/components/PropertyEditWizard.js` — пошаговый визард создания/редактирования (мобильный)
- `src/web/screens/WebPropertiesScreen.js` — список + детали (веб, 2481 строка)
- `src/web/components/WebPropertyEditPanel.js` — боковая панель создания/редактирования (веб)

**DB-триггеры:**
- `trg_auto_set_property_company` → `auto_set_property_company()` — на INSERT, автоматически подставляет `company_id` ❌ НЕ в миграциях (TD-025). Содержит мёртвый код со статусом 'submitted' (TD-052)

**DB constraints:**
- `properties_name_not_blank` — CHECK: name не пустое
- `properties_property_status_check` — CHECK: status IN ('pending', 'approved', 'rejected')
- `properties_type_check` — CHECK: type IN ('house', 'resort', 'condo', 'resort_house', 'condo_apartment')
- `properties_resort_id_fkey` — FK: resort_id → properties(id) ON DELETE CASCADE

## Подмодуль 1: Типы объектов и иерархия (PR-TYPE)

### Типы

Два уровня объектов:

**Самостоятельные (верхний уровень):**
- **House** — отдельный дом, не входит в комплекс
- **Resort** — комплекс, контейнер для домов
- **Condo** — здание, контейнер для квартир

**Дочерние (вложены в контейнер):**
- **Resort House** — дом внутри резорта (`resort_id` = ID резорта)
- **Condo Apartment** — квартира внутри кондо (`resort_id` = ID кондо)

Все объекты хранятся в одной таблице `properties`. Различаются полем `type` и наличием `resort_id`.

### Правила

**PR-TYPE-1.** Тип объекта задаётся при создании и **не может быть изменён**. Дом не может стать резортом, квартира не может стать домом.

**PR-TYPE-2.** Дочерний объект **не может существовать** без родителя. При удалении родителя (резорт/кондо) каскадно удаляются все его дочерние объекты. Обеспечивается FK constraint `properties_resort_id_fkey` с `ON DELETE CASCADE`. UI показывает двойное подтверждение с предупреждением.

**PR-TYPE-3.** Ограничений на количество дочерних объектов в резорте/кондо нет.

**PR-TYPE-4.** *(TD-043, низкий)* Поле `resort_id` используется для связи и с резортами, и с кондо — название не отражает суть. Рекомендуется переименовать в `parent_id`. Масштаб: 78 вхождений в 16 JS-файлах + FK + триггер. Делать только при наличии тестов.

### Наследование дочерних от родителя

| Поле | Наследуется? | Каскадное обновление? |
|---|---|---|
| Район (district) | Да, read-only | Да, `updateResortChildrenDistrict()` |
| Ответственный агент | Да, read-only | Да, `updatePropertyResponsible(cascade=true)` |
| Локация (location_id) | Да, при создании | Нет явного каскада |
| Собственник (owner_id) | Да, при создании | Нет |

### Сортировка дочерних

Дома в резорте / квартиры в кондо сортируются по `code_suffix`: числовые по возрастанию (30, 35, 37), буквенные по алфавиту. Функция `sortByInternalCode()`.

## Подмодуль 2: Создание объекта (PR-CR)

### Обязательные поля

**PR-CR-1.** Обязательные поля при создании: **Название**, **Локация (город)**, **Район**. Все остальные опциональные — дозаполняются позже.

**PR-CR-1.1.** *(TD-044)* На мобильном сейчас обязательно только Название. Нужно добавить проверку Локации и Района (как на вебе).

**PR-CR-1.2.** *(TD-050)* Обязательность Локации проверяется только на клиенте (JS). В БД нет CHECK constraint на `location_id IS NOT NULL`. Нужно добавить серверную валидацию.

### Тип и статус

**PR-CR-2.** Тип объекта задаётся при создании и не меняется (см. PR-TYPE-1).

**PR-CR-3.** Статус при создании:
- Админ → `'approved'` (сразу виден, модерация не нужна)
- Агент → `'pending'` (требует одобрения админа)

**PR-CR-3.1.** Создание объекта агентом возможно **только** при наличии permission `can_add_property` (см. CO-PERM-1, CURSOR_RULES раздел 4.1). *(TD-005: permissions не полностью enforced в UI)*

### Автоматические поля

**PR-CR-4.** `company_id` подставляется автоматически:
- Для агента — из `company_members` через `resolveAgentCompanyId()`
- Для админа — триггером `auto_set_property_company` (если JS-код не подставил)

**PR-CR-5.** `responsible_agent_id` при создании = сам создатель.

**PR-CR-6.** `user_id` при создании = сам создатель. Это фиксация "кто создал", не право доступа (см. CURSOR_RULES CF-001).

### Уникальность кода

**PR-CR-7.** Код объекта (`code`) должен быть **уникальным в рамках компании** для объектов верхнего уровня (house, resort, condo). У разных компаний могут быть одинаковые коды.

**PR-CR-7.1.** Внутренний код (`code_suffix`) дочернего объекта должен быть **уникальным в рамках одного родителя**. Нельзя иметь два дома с одинаковым номером в одном резорте.

**PR-CR-7.2.** *(TD-054)* Уникальность кода сейчас не проверяется ни на клиенте, ни в БД. Нужно добавить: UNIQUE constraint в БД + проверку в UI перед сохранением.

### Визард мобильный — шаги по типам

**PR-CR-8.** Для типов house / resort_house / condo_apartment — **8 шагов**:

| Шаг | Название | Содержимое |
|---|---|---|
| 1 | Info | Название, город/локация, район, код (или code_suffix для дочерних), Google Maps, адрес, собственник (+ доп. собственник для дочерних), ответственный агент (только для админа) |
| 2 | Chars | Спальни, ванные, площадь, этаж (только condo_apartment), расстояние до пляжа/магазина |
| 3 | Desc | Описание |
| 4 | Media | Фотографии (лимит по тарифу: Standard=10, Premium/Korshun=30), видео (URL-ссылки) |
| 5 | Amenities | Удобства (переключатели): бассейн, зал, парковка, интернет, ТВ, стиральная, посудомойка, холодильник |
| 6 | Additional | Кондиционеры, скорость интернета, животные (переключатель), долгосрочная аренда (переключатель) |
| 7 | Pricing | Аренда, депозиты, комиссии, комиссия собственнику (разовая/ежемесячная, сумма или %), коммуналка (электричество, вода с типом, газ, интернет, клининг, выездной клининг) |
| 8 | Comments | Ссылка на сайт объекта, комментарии |

**PR-CR-9.** Для типов resort / condo — **5 шагов**: Info, Chars, Desc, Media, Comments. Без Amenities, Additional, Pricing — контейнеры не сдаются в аренду.

**PR-CR-10.** Отличие Chars для контейнеров:
- Resort: поле "Количество домов" (`houses_count`) вместо спален/ванных/площади
- Condo: поле "Количество этажей" (`floors`) вместо спален/ванных/площади

### Дочерние объекты — особенности создания

**PR-CR-12.** Дочерние объекты (resort_house, condo_apartment) при создании:
- `resort_id` = ID родителя
- Город, район, собственник, ответственный агент — наследуются от родителя (отображаются read-only)
- Используется `code_suffix` вместо `code`
- Доступен дополнительный собственник (`owner_id_2`)
- Расстояние до пляжа/магазина — наследуется от родителя если не задано
- Condo apartment — дополнительное поле "Этаж" (`floor_number`)

### Видимость полей по ролям

**PR-CR-13.** Видимость полей при создании:

| Поле | Админ | Агент |
|---|---|---|
| Собственник | ✅ видно | ✅ видно |
| Ответственный агент | ✅ видно | ❌ скрыто |

Одинаково на вебе и в мобильном приложении.

### Расхождения веб vs мобильный (к исправлению)

| # | Расхождение | TD |
|---|---|---|
| 1 | Обязательные поля: мобильный не проверяет локацию и район | TD-044 |
| 2 | Видео: веб — одна ссылка (`video_url`), мобильный — массив (`videos`) | TD-045 |
| 3 | Валюта: мобильный не хранит `currency` на объекте | TD-046 |
| 4 | Адрес: веб не имеет поля `address` | TD-047 |
| 5 | Собственники: веб не загружает для агента | TD-048 |
| 6 | Ответственный: мобильный показывает агенту (должен быть скрыт) | TD-049 |

### Безопасность

**PR-CR-14.** *(TD-050)* Обязательные поля проверяются только на клиенте. В БД нет constraint на `location_id`. Прямой запрос к API может создать объект без локации.

**PR-CR-15.** *(TD-051)* `createPropertyFull()` принимает произвольные поля через spread (`...updates`). Нет фильтрации допустимых полей перед INSERT. Рекомендация: whitelist разрешённых полей.

**PR-CR-16.** *(TD-052)* Триггер `auto_set_property_company` содержит мёртвый код: пытается установить статус `'submitted'`, которого нет в CHECK constraint. JS-код всегда ставит статус раньше триггера. Мёртвая логика не выполняется, но должна быть удалена.

**PR-CR-17.** *(TD-053)* При удалении объекта фотографии остаются в Supabase Storage. `deleteProperty()` не вызывает `deletePhotoFromStorage()`. Утечка места.

**PR-CR-18.** Лимит фотографий (Standard=10, Premium=30) проверяется на клиенте. Серверная проверка отсутствует. Достаточно проверки в `propertiesService.js` перед сохранением.

## Связь с RBAC

- `can_add_property` — агент может создать объект только при наличии этого permission (CURSOR_RULES 4.1)
- `can_edit_info`, `can_edit_prices` — будут описаны в подмодуле 3 (Редактирование)
- `responsible_agent_id` — определяет видимость объекта для агента (подмодуль 9)
- LOCK-001 — правила удаления (подмодуль 5)

## Связь с другими модулями

| Модуль | Связь |
|---|---|
| **Auth & Session** | `user.isAgentRole` определяет статус при создании (pending vs approved). `user.teamPermissions.can_add_property` разрешает/запрещает создание. |
| **Company & Team** | `company_id` берётся из membership. При деактивации агента `responsible_agent_id → NULL`. `getActiveTeamMembers()` используется для пикера ответственного. |
| **Contacts** | `owner_id` / `owner_id_2` ссылаются на контакты типа "owner". |
| **Bookings** | Бронирования привязаны к `property_id`. При удалении объекта — каскадные последствия (будет описано в подмодуле 5). |
