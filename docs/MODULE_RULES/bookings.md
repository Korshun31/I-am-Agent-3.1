# Module: Bookings

> Документ в процессе написания. Покрыт подмодуль 1 из 7.

## Скоуп

Модуль покрывает: создание, редактирование и удаление бронирований, финансовые расчёты (аренда, депозиты, комиссии), проверку
пересечения дат (overlap), напоминания о заезде и комиссиях, генерацию PDF-подтверждения, видимость бронирований.

**НЕ покрывается:**
- Объекты недвижимости (→ Properties, но связь через `property_id` описана здесь)
- Контакты гостей (→ Contacts, но связь через `contact_id` описана здесь)
- Календарные события пользователя (→ Calendar Events)

**Файлы модуля:**
- `src/services/bookingsService.js` — CRUD, overlap detection
- `src/services/bookingConfirmationService.js` — генерация PDF-подтверждения
- `src/services/bookingRemindersService.js` — напоминания о заезде
- `src/services/commissionRemindersService.js` — напоминания о комиссиях собственнику
- `src/components/AddBookingModal.js` — создание/редактирование (мобильный, 4-шаговый визард)
- `src/screens/BookingDetailScreen.js` — детали бронирования (мобильный)
- `src/screens/BookingCalendarScreen.js` — Gantt-календарь бронирований (мобильный)
- `src/web/components/WebBookingEditPanel.js` — создание/редактирование (веб, панель)
- `src/web/screens/WebBookingsScreen.js` — список + Gantt-календарь (веб)

**DB constraints:**
- FK `bookings_property_id_fkey` → `properties(id)` ON DELETE CASCADE
- FK `bookings_contact_id_fkey` → `contacts(id)` ON DELETE SET NULL
- FK `bookings_company_id_fkey` → `companies(id)` ON DELETE SET NULL
- FK `bookings_agent_id_fkey` → `auth.users(id)` ON DELETE CASCADE (legacy название)
- Нет триггеров на таблице bookings

## Подмодуль 1: Создание бронирования (BK-CR)

### Кто может создать

**BK-CR-1.** Админ — всегда может создать бронирование.

**BK-CR-2.** Агент — только при наличии permission `can_book` (CO-PERM-1, CURSOR_RULES 4.2). Без `can_book` кнопка создания скрыта.

### Обязательные поля

**BK-CR-3.** Обязательные поля при создании:
- **Объект** (`property_id`) — к какому объекту привязано
- **Дата заезда** (`check_in`)
- **Дата выезда** (`check_out`)
- **Контакт (гость) ИЛИ "Клиент собственника"** — одно из двух обязательно

**BK-CR-3.1.** *(TD-073)* Обязательность контакта сейчас **не проверяется** ни на клиенте, ни в БД. Нужно добавить валидацию:
`contact_id IS NOT NULL OR not_my_customer = true`.

**BK-CR-3.2.** Дата выезда должна быть **позже** даты заезда. Проверяется на клиенте.

### "Клиент собственника" (not_my_customer)

**BK-CR-4.** Флаг `not_my_customer` — означает "этого клиента нашёл собственник, не агент". Бронирование без привязки к контакту.
Используется когда собственник сам нашёл арендатора, а пользователь просто фиксирует бронь для учёта.

**BK-CR-5.** Если "Клиент собственника" включён — финансовые поля и напоминания **скрываются**. Бронирование сохраняется сразу после
 выбора дат. Одинаково на вебе и мобильном.

**BK-CR-5.1.** *(TD-083)* На вебе сейчас все поля видны всегда, даже при "Клиент собственника". Нужно скрывать как на мобильном.

### Проверка пересечения дат (overlap detection)

**BK-CR-6.** Перед созданием проверяется: нет ли другого бронирования на тот же объект с пересекающимися датами. Если есть — ошибка
`BOOKING_CONFLICT`. Один объект не может быть сдан двум клиентам одновременно.

**BK-CR-7.** На мобильном проверка выполняется **дважды**:
1. На клиенте — занятые даты подсвечены в календаре, при выборе пересечения — мгновенная ошибка
2. На сервере — `checkBookingConflict()` перед INSERT

**BK-CR-7.1.** *(TD-072)* На вебе проверка только на сервере. Нужно добавить визуальное отображение занятых дат при выборе (как на
мобильном).

### Предзаполнение цен из объекта

**BK-CR-8.** При создании **нового** бронирования (не при редактировании) автоматически подтягиваются цены из карточки объекта:
- Аренда в месяц ← `property.price_monthly`
- Депозит бронирования ← `property.booking_deposit`
- Страховой депозит ← `property.save_deposit`
- Комиссия агента ← `property.commission`
- Комиссия собственнику (разовая) ← `property.owner_commission_one_time`
- Комиссия собственнику (ежемесячная) ← `property.owner_commission_monthly`

### Автоматический расчёт общей стоимости

**BK-CR-9.** Общая стоимость рассчитывается **помесячно** с учётом реального количества дней в каждом месяце:
1. Полные месяцы (от числа до того же числа) = аренда в месяц × количество полных месяцев
2. Оставшиеся дни = аренда в месяц / количество дней в том конкретном месяце × количество оставшихся дней
3. Общая стоимость = сумма

Примеры (аренда 30 000):
- 20 фев → 20 мар = 1 месяц = 30 000
- 20 фев → 25 мар = 1 месяц + 5 дней в марте (31 день) = 30 000 + 30 000/31×5 = 34 839
- 15 янв → 28 фев = 1 месяц + 13 дней в феврале (28 дней) = 30 000 + 30 000/28×13 = 43 929

**BK-CR-9.1.** *(TD-080)* Текущая формула неправильная: `цена × ночей / 30`. Не учитывает реальное количество дней в месяце. Нужно
переписать на обеих платформах.

### Помесячная разбивка

**BK-CR-10.** Под полем "Аренда в месяц" — переключатель **"Помесячная разбивка"**. По умолчанию выключен.

**BK-CR-10.1.** Когда **выключен**: одна цена на весь период, общая стоимость считается автоматически.

**BK-CR-10.2.** Когда **включён**: появляется список строк — по одной на каждый период (месяц или остаток). Все строки заполняются
базовой ценой. Пользователь меняет те месяцы где цена отличается (сезонные колебания). Последний период (если неполный) —
автоматически рассчитывается пропорционально. Итого пересчитывается мгновенно.

**BK-CR-10.3.** При выключении переключателя — возвращается к единой цене, разбивка сбрасывается.

**BK-CR-10.4.** *(TD-082)* Помесячная разбивка сейчас не реализована. Нужно: новое поле `monthly_breakdown` (JSONB) в таблице
`bookings`, UI на обеих платформах, связь с PDF-подтверждением (раздел "Оплата").

### Валюта

**BK-CR-11.** При создании бронирования цены показываются в валюте объекта. Сохраняется в валюте, выбранной в настройках
пользователя. Пользователь может переключить валюту и перезаполнить цены вручную. В будущем: автоматическая конвертация по мировым
курсам.

### Визард мобильный — 4 шага

**BK-CR-12.** Мобильный — пошаговый визард:

| Шаг | Что заполняется |
|---|---|
| 1. Гость | Контакт (или "Клиент собственника"), паспорт, взрослые, дети, животные, комментарии |
| 2. Даты | Календарь с занятыми днями, даты заезда/выезда, время заезда/выезда (по умолчанию 14:00/12:00) |
| 3. Финансы | Аренда, общая стоимость (авто), депозиты, комиссии |
| 4. Напоминания и фото | Напоминания (1/3/7/30 дней), фотографии |

Если "Клиент собственника" — после шага 2 сразу сохранение, шаги 3-4 пропускаются.

### Веб-панель

**BK-CR-13.** Веб — боковая панель со всеми полями. Секции: объект, гость, даты, финансы, доп. информация, фото.

### Автоматические поля

**BK-CR-14.** При создании автоматически заполняются:
- `user_id` = создатель
- `company_id` = из объекта (`property.company_id`)
- `currency` = из настроек пользователя

### Безопасность и RLS

**BK-CR-15.** RLS политики на `bookings`:

| Политика | Кто | Что может |
|---|---|---|
| `owner full access to company` | Админ | Всё |
| `agent reads assigned property bookings` | Агент | Читает бронирования объектов где он ответственный |
| `agent read own` | Агент | Читает свои бронирования |
| `agent insert own` | Агент | Создаёт |
| `agent update own` | Агент | Редактирует только свои |
| `agent delete own` | Агент | Удаляет только свои |

**BK-CR-15.1.** *(TD-084)* Политика `company member read` даёт агенту доступ ко **всем** бронированиям компании. Нарушает правило:
агент видит только бронирования своих объектов. Нужно убрать или ограничить.

**BK-CR-16.** Когда на объект агента бронирование создал админ — агент видит факт бронирования, но **не видит имени клиента**. Видит
 что бронирование "на Компанию".

**BK-CR-16.1.** *(TD-085)* Сейчас агент видит все данные бронирования, включая имя клиента. Нужно скрывать контактные данные если
бронирование создано не агентом.

### Напоминания

**BK-CR-17.** *(TD-079)* `bookingRemindersService` и `commissionRemindersService` используют устаревший формат trigger для
expo-notifications (тот же баг что был в `calendarRemindersService`). Напоминания о заезде и комиссии **не работают** на мобильном.

**Все правила этого подмодуля применяются одинаково на вебе и мобильном.**

## Подмодуль 2: Редактирование бронирования (BK-EDIT)

### Кто может редактировать

**BK-EDIT-1.** Админ — редактирует любое бронирование компании. При редактировании бронирования агента — `booking_agent_id` **не
меняется** автоматически. Бронирование остаётся "на агента".

**BK-EDIT-2.** Агент — редактирует бронирование если:
- Есть permission `can_book`
- `booking_agent_id = его id` (он ответственный за это бронирование)

Не важно кто создал — важно кто назначен ответственным.

**BK-EDIT-3.** Если бронирование "на Компанию" (`booking_agent_id = NULL` или другой агент) — агент **не может** редактировать.
Видит только как "Клиент компании" (BK-CR-16).

### Ответственный за бронирование (booking_agent_id)

**BK-EDIT-4.** Новое поле `booking_agent_id` определяет кто отвечает за бронирование:
- `NULL` — бронирование "на Компанию" (ответственный — админ)
- `UUID агента` — бронирование на конкретного агента

**BK-EDIT-5.** Как назначается:
- Агент создаёт → `booking_agent_id = он сам` (автоматически)
- Админ создаёт → `booking_agent_id = NULL` (по умолчанию) или выбирает агента из пикера
- Админ редактирует → может изменить ответственного через пикер

**BK-EDIT-6.** Поле "Ответственный за бронирование" видно **только админу** при создании и редактировании. Агент **не видит** это
поле. Одинаково на вебе и мобильном.

**BK-EDIT-7.** *(TD-086)* `booking_agent_id` сейчас не существует в БД. Нужно:
- Новая колонка `booking_agent_id` (UUID, nullable, FK → auth.users ON DELETE SET NULL)
- Backfill: `UPDATE bookings SET booking_agent_id = user_id` (для существующих)
- Обновить RLS: `agent read own`, `agent update own`, `agent delete own` — заменить `user_id` на `booking_agent_id`
- Оставить `agent reads assigned property bookings` (чтение через properties.responsible_agent_id) — агент видит бронирования своих
объектов для информации, но без имени клиента если `booking_agent_id != его id`
- Убрать `company member read` (TD-084)
- Пикер ответственного — только для админа

### Что отличается от создания

**BK-EDIT-8.** При редактировании:
- Цены **не** предзаполняются из объекта — берутся из сохранённого бронирования
- Общая стоимость **не** пересчитывается автоматически (пользователь контролирует вручную)
- Overlap проверяется повторно с исключением текущего бронирования (`excludeId`)

**BK-EDIT-9.** На мобильном при редактировании отменяются старые напоминания перед обновлением (`cancelBookingReminders` +
`cancelCommissionReminders`), потом регистрируются новые.

**BK-EDIT-9.1.** *(TD-077)* На вебе отмена старых напоминаний не выполняется.

### Сравнение веб vs мобильный

| Аспект | Мобильный | Веб | Совпадает? |
|---|---|---|---|
| Форма редактирования | Тот же визард (AddBookingModal) | Та же панель (WebBookingEditPanel) | ✅ |
| Отмена старых напоминаний | ✅ | ❌ (TD-077) | ⚠️  |
| Пикер ответственного (booking_agent_id) | ❌ Не реализован | ❌ Не реализован | ⚠️  TD-086 |

**Все правила этого подмодуля применяются одинаково на вебе и мобильном.**

## Подмодуль 3: Удаление бронирования (BK-DEL)

### Кто может удалить

**BK-DEL-1.** Админ — удаляет любое бронирование компании.

**BK-DEL-2.** Агент — удаляет бронирование если:
- `booking_agent_id = его id` (он ответственный за это бронирование)
- Есть permission `can_delete_booking`

Без `can_delete_booking` — кнопка удаления **скрыта**, даже для своих бронирований.

**BK-DEL-2.1.** *(TD-087)* Переключатель `can_delete_booking` отсутствует в UI настроек агента (обе платформы). В коде
`deleteBooking()` permission не проверяется. Нужно добавить переключатель + проверку.

### Что происходит при удалении

**BK-DEL-3.** При удалении бронирования:
1. Отменяются напоминания о комиссиях (`cancelCommissionReminders`)
2. Отменяются напоминания о заезде (`cancelBookingReminders`)
3. Удаляется запись из `bookings`
4. `broadcastChange('bookings')`

**BK-DEL-4.** Каскадных последствий нет — ни одна таблица не ссылается на `bookings` через FK. Контакт, объект, компания — не
затрагиваются.

### Сравнение веб vs мобильный

| Аспект | Мобильный | Веб | Совпадает? |
|---|---|---|---|
| Проверка `can_delete_booking` | ❌ Не проверяется | ❌ Не проверяется | ⚠️  TD-087 |
| Отмена напоминаний перед удалением | ✅ `cancelCommissionReminders` | Нужно проверить | ⚠️  |

**Все правила этого подмодуля применяются одинаково на вебе и мобильном.**

## Подмодуль 4: Финансы бронирования (BK-FIN)

### Видимость финансов

**BK-FIN-1.** Агент **всегда** видит финансовые поля своего бронирования (`booking_agent_id = его id`). Отдельного permission на
видимость финансов нет.

**BK-FIN-1.1.** *(TD-088)* `can_see_financials` — удалить из проекта. На вебе комиссии собственнику обёрнуты в `canSeeFinancials` —
нужно убрать эту обёртку. Удалить из CO-PERM-1, из CURSOR_RULES раздел 4.2, из кода.

### Расчёт общей стоимости

**BK-FIN-2.** Формула расчёта — помесячная (BK-CR-9). Должна **совпадать** между:
- Формой создания/редактирования (`computeTotalPrice`)
- PDF-подтверждением (`buildPaymentPlanRows`)

**BK-FIN-2.1.** *(TD-080)* Сейчас формулы **не совпадают**: формы считают `цена × ночей / 30`, PDF считает помесячно с реальными
днями. Нужно исправить формы.

### Помесячная разбивка

**BK-FIN-3.** Переключатель "Помесячная разбивка" (BK-CR-10) позволяет задать разную стоимость по месяцам. Данные хранятся в
`monthly_breakdown` (JSONB). Когда реализовано — PDF-подтверждение (`buildPaymentPlanRows`) берёт суммы из `monthly_breakdown`
вместо единой `price_monthly`.

### Комиссия собственнику — %/сумма

**BK-FIN-4.** Комиссия собственнику (разовая и ежемесячная) может быть задана как **сумма** или как **процент** от аренды в месяц.
Переключатель %/сумма + автоматический расчёт суммы из процента.

**BK-FIN-4.1.** *(TD-089)* На мобильном (`AddBookingModal`) переключатель %/сумма и расчёт **отсутствуют** — просто текстовые поля.
На вебе (`WebBookingEditPanel`) — **реализовано**. Нужно добавить на мобильный по примеру `PropertyEditWizard` (объекты).

### Формула "Итого" в PDF-подтверждении

**BK-FIN-5.** В PDF-подтверждении:
- Итого = общая стоимость аренды + страховой депозит
- Оплачено = депозит бронирования
- Остаток = итого - оплачено

### Предзаполнение

**BK-FIN-6.** При создании нового бронирования цены предзаполняются из карточки объекта (BK-CR-8). При редактировании — из
сохранённого бронирования.

### Валюта

**BK-FIN-7.** Валюта бронирования (BK-CR-11): показывается в валюте объекта, сохраняется в валюте настроек пользователя.

### Сравнение веб vs мобильный

| Переключатель %/сумма для комиссии | ❌ Нет | ✅ Есть | ⚠️  TD-089 |
| Расчёт суммы из процента | ❌ Нет | ✅ Есть | ⚠️  TD-089 |
| `canSeeFinancials` обёртка | Нет | ✅ Есть (нужно убрать) | ⚠️  TD-088 |

**Все правила этого подмодуля применяются одинаково на вебе и мобильном.**

## Связь с RBAC

- `can_book` — агент может создать бронирование только при наличии (CURSOR_RULES 4.2)
- `can_delete_booking` — агент может удалить бронирование только при наличии (BK-DEL-2)
- `booking_agent_id` — определяет кто отвечает за бронирование и может его редактировать/удалять (BK-EDIT-4)
- `can_see_financials` — **удалить из проекта** (TD-088). Агент всегда видит финансы своих бронирований.

## Связь с другими модулями

| Модуль | Связь |
|---|---|
| **Auth & Session** | `user.teamPermissions.can_book` разрешает/запрещает создание. |
| **Properties** | `property_id` — объект бронирования. Цены предзаполняются. При удалении объекта бронирования удаляются каскадно (PR-DEL-5). |
| **Contacts** | `contact_id` — гость. При удалении контакта `contact_id → NULL` (FK ON DELETE SET NULL). |
| **Company & Team** | `company_id` берётся из объекта. При деактивации агента — бронирования остаются (user_id сохранён). |
