# Сквозные правила: RBAC (Управление доступом)

> Эти правила действуют во ВСЕХ модулях проекта. При противоречии с правилами конкретного модуля — этот документ имеет приоритет.

## RBAC-1. Роли и разрешения

У каждого пользователя есть **роль**. Для каждой роли определены:
- **Жёсткие правила** — что роль точно НЕ может делать. Зашиты в систему. Никто не может их изменить.
- **Настраиваемые разрешения** — переключатели которые включает/выключает тот кто пригласил. Можно менять в любой момент.

При добавлении новой роли — задаются оба слоя. Код не меняется, потому что проверяет не роль, а разрешения.

### Текущие роли

**Admin** — зарегистрировался сам, владелец компании. Все разрешения включены. Все жёсткие правила в его пользу — видит всё, может всё.

**Agent** — приглашён админом в команду. Часть разрешений настраивается админом через переключатели. Часть — жёсткие правила:
- Не видит поле "Ответственный агент"
- Не может одобрять/отклонять объекты
- Не может менять ответственного
- Не может удалить одобренный объект (LOCK-001)
- Не может редактировать/удалять чужие бронирования
- Не может управлять локациями (только чтение + добавление районов)
- Не может приглашать/удалять членов команды
- Не может менять права других агентов
- Видит только свои объекты и бронирования

**Обычный пользователь** (без команды) — зарегистрировался, но не создавал компанию и не был приглашён. Все разрешения включены (как админ, но без команды).

### Планируемые роли

**Owner (независимый)** — собственник недвижимости который сам зарегистрировался. Свои объекты, может расшарить ограниченный доступ к ним другим пользователям (через QR-код или
 ссылку, как Airbnb с ключом к календарю).

**Owner (приглашённый)** — собственник которого пригласила компания. Не платит за CRM, пользуется бесплатно. Его данные привязаны к компании. Делиться с другими не может.

При реализации owner — задаются жёсткие правила и настраиваемые разрешения для каждого типа. Код не переписывается.

## RBAC-2. Запрет на проверку роли в коде

В коде **запрещено** проверять роль напрямую:
- ❌ `if (isAgentRole)` — спрашивает "кто ты?" вместо "что тебе разрешено?"
- ❌ `if (isAdminRole)` — то же самое
- ❌ `!!user.teamMembership` — старый способ, будет удалён
- ❌ `user.role` — мусорное поле (TD-001)
- ❌ `user.workAs === 'company'` — для UI режима, не для проверки прав

В коде **разрешено** проверять только разрешения:
- ✅ `if (user.permissions.can_add_property)` — "разрешено ли добавлять объекты?"
- ✅ `if (user.permissions.can_book)` — "разрешено ли бронировать?"

Вместо вопроса "кто ты?" — вопрос "что тебе разрешено?". Не важно какая роль — важно что конкретно разрешено.

> **Примечание:** текущий код использует `isAgentRole` / `isAdminRole` повсеместно. Переход на permission-based проверки — постепенный процесс. При написании нового кода и при
рефакторинге существующего — использовать только permissions.

## RBAC-3. Настраиваемые разрешения

Текущий список переключателей:

| Разрешение | Что даёт | Если выключено |
|---|---|---|
| `can_add_property` | Может добавлять объекты | Кнопка "+" скрыта. Объекты попадают только при назначении ответственным |
| `can_edit_info` | Может редактировать информацию объекта | Изменения идут через черновик на проверку |
| `can_edit_prices` | Может редактировать цены объекта | Изменения цен идут через черновик |
| `can_book` | Может создавать бронирования | Кнопка создания скрыта |
| `can_delete_booking` | Может удалять бронирования | Кнопка удаления скрыта |

**Удалены из проекта:**
- `can_see_financials` (TD-088) — агент всегда видит финансы своих бронирований
- `can_manage_clients` (TD-102) — агент всегда работает со своими контактами

Новые разрешения для owner будут добавлены при реализации роли owner.

## RBAC-4. Двойная защита данных

Видимость данных определяется на **двух уровнях**:

1. **Первая стена — база данных (RLS).** Когда приложение запрашивает данные, база сама отсекает чужое. Админ получает все данные компании. Агент — только свои. Приложение даже
 не знает что другие данные существуют.

2. **Вторая стена — приложение.** Дополнительная проверка в коде. Если первая стена пропустила лишнее — приложение отфильтрует. Это страховка.

Две стены лучше чем одна. Если одна сломается — вторая защитит.

## RBAC-5. Невидимость недоступных действий

Если пользователь **не может** выполнить действие — кнопка **не показывается** вообще. Не серая, не заблокированная — полностью невидимая. Человек не должен видеть то чего не
может сделать.

## Связь с модулями

Эти правила применяются в:
- **Properties**: PR-CR-3.1 (can_add_property), PR-EDIT-3/4 (can_edit_info/prices), PR-DEL-2 (LOCK-001), PR-VIS-1/2 (видимость), PR-CR-13/PR-EDIT-6/PR-RESP-6 (ответственный
скрыт)
- **Bookings**: BK-CR-2 (can_book), BK-DEL-2 (can_delete_booking), BK-VIS-1/2 (видимость), BK-EDIT-6 (booking_agent_id скрыт)
- **Contacts**: CT-VIS-1/2 (видимость), CT-ED-2/5 (только свои)
- **Calendar Events**: CE-VIS-1 (только свои)
- **Company & Team**: CO-PERM-1/2 (настройка permissions)
- **Auth & Session**: AU-PROFILE-2 (канонические поля), AU-PROFILE-6 (направление на permission-based)
