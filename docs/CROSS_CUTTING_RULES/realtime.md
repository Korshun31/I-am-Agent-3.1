# Сквозные правила: Realtime (Обновление данных)

> Эти правила действуют во ВСЕХ модулях проекта.

## RT-1. Два действия после каждого изменения

После **любого** изменения данных (создание, редактирование, удаление) — обязательно:
1. Вызвать `refreshX()` — обновить данные у текущего пользователя
2. Вызвать `broadcastChange('table')` — сообщить другим участникам компании

Без первого — текущий пользователь не увидит изменения. Без второго — другие участники не узнают.

## RT-2. broadcastChange — задача сервиса

Сервисы (`propertiesService`, `bookingsService` и т.д.) **уже** вызывают `broadcastChange` внутри себя. Экран не должен вызывать `broadcastChange` отдельно — это задача
сервиса.

## RT-3. Broadcast — лёгкий сигнал

`broadcastChange` — лёгкий сигнал через WebSocket. Не передаёт данные. Только говорит "таблица X изменилась". Получатель сам решает что обновить.

Это **не** подписка на изменения в базе (`postgres_changes`) — та нагружает сервер. Broadcast — безопасный, лёгкий.

**ЗАПРЕЩЕНО** (из CURSOR_RULES 2.2):
- `supabase.channel().on('postgres_changes')` — только где уже есть
- `window.addEventListener` / `window.dispatchEvent`
- `setInterval` / polling
