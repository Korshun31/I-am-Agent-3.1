# Сквозные правила: State Management

> Эти правила действуют во ВСЕХ модулях проекта.

## SM-1. Не копировать данные

Если данные уже есть в общем хранилище (`AppDataContext` на мобильном) — использовать оттуда. Не создавать копию в отдельной переменной.

Копия устаревает. Оригинал обновляется, а копия — нет. Пользователь видит старые данные.

Пример бага который мы починили: `customEvents` был копией `calendarEvents`. Копия устаревала — новые события не появлялись без перезагрузки.

**Правило:** если нужны данные — бери из источника. Не клонируй.

## SM-2. После изменения — обновить через хранилище

После создания, редактирования или удаления данных — вызвать обновление из общего хранилища (`refreshProperties()`, `refreshBookings()` и т.д.) + отправить сигнал другим
участникам (`broadcastChange()`).

Не пытаться обновить UI вручную подставляя данные. Пусть хранилище загрузит свежие и UI обновится автоматически.

✅ Правильно: создал бронирование → `refreshBookings()` → список обновился сам.

❌ Неправильно: создал бронирование → вручную добавил в локальный массив → массив рассинхронизировался с базой.

## SM-3. Не использовать устаревшие переменные после await

Когда внутри функции нужно дождаться обновления данных (`await`) и потом использовать эти данные — нельзя использовать переменную которая была получена **до** ожидания.

Пока ты ждал — переменная устарела. Новые данные уже в хранилище. Но старая переменная об этом не знает.

**Правило:** после `await` — брать свежие данные из хранилища, а не из старой переменной.

## Связь с модулями

Эти правила применяются во всех модулях с CRUD операциями: Properties, Bookings, Contacts, Calendar Events.
