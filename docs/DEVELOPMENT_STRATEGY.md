# 📐 Стратегия разработки I am Agent

> Документ создан 2026-04-11. Описывает методологию работы над проектом и порядок дальнейшего развития.

## Основная концепция: specification-first

С 2026-04-11 проект следует подходу **specification-first development**: сначала пишем правила для каждого модуля (как чертежи здания), потом реализуем, рефакторим и покрываем тестами.

**Метафора:** Когда строишь дом без чертежей, каждое следующее изменение становится всё дороже — приходится ломать уже сделанное. С чертежами наоборот: один раз думаешь сложно, потом строишь быстро и без переделок. В коде то же самое — изменение спецификации на бумаге стоит копейки, изменение готового кода стоит "сломать + построить заново".

**Что это значит на практике:**

1. **Прежде чем рефакторить** какой-либо модуль — у него должны быть зафиксированные правила.
2. **Прежде чем писать тесты** — должны быть правила, ОТ КОТОРЫХ эти тесты пишутся. Тест без правила не имеет смысла.
3. **Прежде чем добавлять новый модуль** — пишутся его правила и проверяется непротиворечивость с существующими.
4. **При изменении поведения существующего модуля** — сначала обновляются правила, потом код.

## Связь с техническим долгом и рефакторингом

**Технический долг** — это разница между тем, как код устроен сейчас, и тем, как он должен быть устроен по правилам. Без правил эту разницу нельзя измерить, поэтому технический долг становится неуправляемым.

**Рефакторинг** — это работа по сокращению этой разницы (приведение кода в соответствие с правилами) БЕЗ изменения внешнего поведения.

**Тесты** — это страховка, которая позволяет рефакторить безопасно. Тесты проверяют что внешнее поведение не изменилось, даже если внутренняя структура кода радикально переписана.

Связка работает так:
- Правила → определяют желаемое поведение
- Тесты → фиксируют текущее (или желаемое) поведение
- Рефакторинг → меняет реализацию, не меняя поведения
- Технический долг → метрика расхождения между реальностью и правилами

## Структура хранения правил

```
docs/
├── DEVELOPMENT_STRATEGY.md          ← этот файл (мастер-документ)
├── MODULE_RULES/
│   ├── README.md                    ← список модулей и навигация
│   ├── auth.md                      ← Auth & Session
│   ├── company.md                   ← Company & Team
│   ├── properties.md                ← Properties (Real Estate)
│   ├── bookings.md                  ← Bookings
│   ├── contacts.md                  ← Contacts
│   ├── calendar_events.md           ← Calendar Events
│   ├── notifications.md             ← Notifications & Reminders
│   ├── sync.md                      ← Sync & Realtime
│   ├── i18n.md                      ← i18n & Settings
│   ├── statistics.md                ← Statistics & Dashboard
│   └── ui_infrastructure.md         ← UI Infrastructure
└── CROSS_CUTTING_RULES/
    ├── rbac.md                      ← Правила работы с правами доступа
    ├── state_management.md          ← Правила работы с React state и контекстом
    ├── i18n_rules.md                ← Правила добавления переводов
    ├── realtime.md                  ← Правила работы с broadcast и refresh после мутаций
    ├── error_handling.md            ← Правила показа и обработки ошибок
    └── validation.md                ← Правила валидации пользовательского ввода
```

**Принципы:**
- Один файл = один модуль (легко читать, легко править, минимум merge-конфликтов)
- Сквозные правила — отдельная директория, чтобы не дублировать одно и то же в каждом модуле
- Каждый файл — самостоятельный; читатель не должен прыгать между файлами для понимания одного правила

## Карта модулей проекта

Полная карта (составлена 2026-04-11). Уровень критичности: 🔴 критично (деньги, права, потеря данных), 🟡 важно (UX, заметные регрессии), 🟢 низкий приоритет.

### 1. Authentication & Session 🔴
**Файлы:** `authService.js`, `Login.js`, `Registration.js`
**Подмодули:** регистрация, логин (email/пароль), Google OAuth, Facebook OAuth, восстановление пароля, загрузка профиля при старте, logout.

### 2. Company & Team 🔴
**Файлы:** `companyService.js`, `WebAccountScreen.js`, `AccountScreen.js`
**Подмодули:** создание компании, приглашение участника, принятие приглашения, удаление участника, изменение прав, передача владения.

### 3. Properties (Real Estate) 🔴
**Файлы:** `propertiesService.js`, `PropertyDetailScreen.js`, `RealEstateScreen.js`, `WebPropertiesScreen.js`
**Подмодули:** создание (как user → pending), создание (как owner → approved), approval/rejection, редактирование, удаление, загрузка фото, привязка локации, фильтрация по правам, видимость pending.

### 4. Bookings 🔴
**Файлы:** `bookingsService.js`, `BookingDetailScreen.js`, `BookingCalendarScreen.js`, `WebBookingsScreen.js`
**Подмодули:** создание, редактирование, удаление, расчёт overlap, расчёт финансов, расчёт комиссии собственнику, привязка контакта, видимость (agent vs admin), подтверждение брони (PDF/ссылка), Gantt-календарь, мобильный календарь.

### 5. Contacts 🟡
**Файлы:** `contactsService.js`, `ContactsScreen.js`, `ContactDetailScreen.js`, `WebContactsScreen.js`
**Подмодули:** создание (client/owner), редактирование (с блокировкой смены типа), удаление (с проверкой связей), видимость, permissions агента.

### 6. Calendar Events 🟡
**Файлы:** `calendarEventsService.js`, `AgentCalendarScreen.js`, `AddCalendarEventModal.js`, `WebAddCalendarEventModal.js`
**Подмодули:** создание события (вручную), редактирование, удаление, отображение в дне, recurring events, toggle "выполнено".
**Заметки:** правила черновика обсуждены 2026-04-11 (CE-CR-1..7, UX-1..5, RM-1..4, ST-1..3). Известный UX gap: нельзя перенести событие на другую дату.

### 7. Notifications & Reminders 🟡
**Файлы:** `calendarRemindersService.js`, `bookingRemindersService.js`, `commissionRemindersService.js`, `notificationsService.js`
**Подмодули:** локальное напоминание (event), локальное напоминание (booking), напоминание о комиссии, отмена при удалении, in-app бейдж, permissions request, Android channel.

### 8. Sync & Realtime 🔴
**Файлы:** `companyChannel.js`, `dataUploadService.js`
**Подмодули:** broadcast канал компании, триггер обновления после мутации, offline sync, восстановление сессии.

### 9. i18n & Settings 🟡
**Файлы:** `LanguageContext.js`, `translations.js`
**Подмодули:** переключение языка (en/th/ru), sync языка из БД, переключение валюты, форматирование дат/чисел.

### 10. Statistics & Dashboard 🟡
**Файлы:** `StatisticsScreen.js`, `WebDashboardScreen.js`
**Подмодули:** месячная выручка, комиссии за период, top-N объектов, графики dashboard, фильтрация по правам.

### 11. UI Infrastructure 🔴
**Файлы:** `App.js`, `MainScreen.js`, `WebMainScreen`, `AppDataContext.js`, `Preloader.js`
**Подмодули:** platform routing, AppDataContext refresh, preloader, bottom tabs, web sidebar layout, scroll positions.

## Этапы работы

### Этап 1: Чертежи модулей (правила)
Идём по таблице сверху вниз, по одному модулю за сессию:
1. Auth & Session (базовый, всё на нём держится)
2. Company & Team (определяет права)
3. Properties (главный объект данных)
4. Bookings (самая сложная бизнес-логика)
5. Contacts
6. Calendar Events (уже частично черновик)
7. Notifications & Reminders
8. Sync & Realtime
9. i18n & Settings
10. Statistics
11. UI Infrastructure

### Этап 2: Сквозные правила
- RBAC (как проверять права)
- State management (как работать с контекстом, без stale closure)
- i18n (как добавлять новые тексты в 3 языка)
- Realtime (как правильно обновлять данные после мутаций)
- Error handling (Alert vs inline)
- Validation (где и как)

### Этап 3: Проверка непротиворечивости
- Модульные vs сквозные правила
- Правила vs реальный код
- Правила vs существующая документация (`CURSOR_RULES.md`, `SYSTEM_ARCHITECTURE.md`, `APP_MAP_WEB.md`)
- Правила vs backlog в `CONTEXT_FOR_AI.md`

### Этап 4: Аудит технического долга и orphan/dormant кода
- Переструктурировать TD-001..TD-013 в `CURSOR_RULES.md` с категориями и приоритетами
- Найти модули-сироты (orphans — неиспользуемый код)
- Найти спящий код (dormant — формально живой, но не работает по назначению)
- Привязать каждый пункт TD к конкретному модулю

### Этап 5: Тесты для критичных подмодулей
- Поставить Jest
- Покрыть тестами критичные (🔴) подмодули в порядке: Bookings → Properties → Auth → Company → Sync
- Один тест = одно правило = одно ожидаемое поведение

### Этап 6: Постепенный рефакторинг
- Маленькими шагами, один модуль за раз
- После каждого шага — тесты должны быть зелёными
- Никакого "большого переписывания с нуля" — только инкрементальные правки

## Формат правил в файлах модулей

Каждый файл правил модуля должен содержать:

```markdown
# Module: <название>

## Скоуп
Что входит в модуль, что НЕ входит (со ссылками на смежные модули).

## Подмодули
Список с короткими описаниями.

## Правила бизнес-логики
Пронумерованные правила вида "<префикс модуля>-<подмодуль>-<номер>".
Пример: CE-CR-1 = Calendar Events / Create / правило 1.

## Правила UX
То же, но про поведение в интерфейсе.

## Контракты функций (опционально)
Входы, выходы, обработка ошибок.

## Связь с RBAC
Какие проверки прав применяются.

## Связь с другими модулями
Зависимости от других модулей и наоборот.

## Известные пробелы и TD
Что НЕ соответствует правилам сейчас и почему.
```

## Как добавлять новый модуль

Когда нужна новая фича/модуль, процесс:

1. Создать файл `docs/MODULE_RULES/<новый_модуль>.md` с правилами по шаблону выше
2. Обсудить (с собой, с командой, с ИИ-помощником) — нет ли противоречий с существующими модулями
3. Обновить навигацию в `docs/MODULE_RULES/README.md`
4. Только после этого писать код
5. По мере реализации — уточнять правила, если обнаружились пробелы
6. После завершения — добавить тесты для критичных правил
7. Пометить в `CURSOR_RULES.md` как новый модуль

## Что НЕ покрывается правилами

- Чисто визуальные мелочи (цвета, отступы, шрифты — они в `CLAUDE.md` секция Conventions и в коде через константы)
- Внутренние детали реализации, которые могут быть пересмотрены без последствий
- Временные хаки, помеченные `TODO` или `FIXME`
