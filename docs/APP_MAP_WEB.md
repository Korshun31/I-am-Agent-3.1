### 2.4 Правила управления собственниками и ответственным

**Собственники:**

| Тип объекта | owner_id | owner_id_2 |
|---|---|---|
| Отдельный дом (standalone) | ✅ редактируется | — |
| Резорт / Кондо (родитель) | ✅ редактируется | — |
| Дом внутри резорта (child) | 🔒 унаследован от родителя (только чтение) | ✅ редактируется |
| Апартамент внутри кондо (child) | 🔒 унаследован от родителя (только чтение) | ✅ редактируется |

**Ответственный агент:**

| Тип объекта | Поведение |
|---|---|
| Родительский резорт / кондо | Редактируется в Edit Panel. Изменение каскадируется на **все** дочерние объекты автоматически. |
| Дочерний дом / апартамент | 🔒 Унаследован от родителя. Ручное изменение заблокировано (пикер скрыт). |
| Отдельный дом (standalone) | Редактируется в Edit Panel. Каскада нет. |

> Новый дочерний объект (`create-unit`) автоматически наследует `responsible_agent_id` родителя в момент создания.

## 6.1 Review-flow — снят 2026-04-30 (этап 2)

Модерация выпилена. Уведомления `property_submitted`, `edit_submitted`, `property_approved`, `property_rejected`, `edit_approved`, `edit_rejected`, `price_*` удалены из кода и БД. Кнопки «Одобрить / Отклонить» убраны из веб-панели уведомлений. `DiffModal`, правая панель в режиме readOnly с блоком Approve/Reject — не используются.

Вместо модерации админу приходят информационные уведомления `property_created` и `booking_created` когда агент добавляет объект или бронирование.

## 6.2 Статусы объекта — сняты 2026-04-30 (этап 2)

Колонка `properties.property_status` физически остаётся в БД до этапа 3 (cleanup-миграция), но не используется ни кодом, ни RLS. Все объекты сразу `approved` через DB-default. Бейджи «На проверке» / «Отклонён» в UI убраны.

## 6.3 История отклонений — снята 2026-04-30 (этап 2)

Таблица `property_rejection_history` физически остаётся до этапа 3 (cleanup-миграция), но не пишется и не читается. Блок «История отклонений» в `PropertyDetail` удалён.

## Phase 1: Унификация словаря ролей (2026-03-30)

> Совместимость: все существующие экраны продолжают работать без изменений.

### Новые поля профиля пользователя

| Поле | Тип | Пример |
|------|-----|--------|
| `user.plan` | string | `'standard'` / `'premium'` / `'korshun'` |
| `user.teamRole` | string\|null | `'agent'` / `'admin'` / `null` |
| `user.isAgentRole` | boolean | `true` если role=agent в company_members |
| `user.isAdminRole` | boolean | `true` если владелец компании |

### Правила гардов (рекомендация Phase 1 → обязательно Phase 2)

- Миграция гардов с `isAgent` на `isAgentRole` завершена в этапе 2 (2026-04-30): все экраны переведены на permission-based проверки `can_manage_property` / `can_manage_bookings`. Старые гарды `canEditInfo`, `canEditPrices`, `canBook`, `canDeleteBooking`, `canSeeFinancials`, `canManageClients` выпилены.
- Legacy: `isAgent = !!user?.teamMembership` продолжает работать (эквивалентно `isAgentRole`).
- Роль `worker` удалена: оставлены только `admin` и `agent`.
