### 2.4 Правила управления собственниками и ответственным

**Собственники:**

| Тип объекта | owner_id | owner_id_2 |
|---|---|---|
| Отдельный дом (standalone) | ✅ редактируется | — |
| Резорт / Кондо (родитель) | ✅ редактируется | — |
| Дом внутри резорта (child) | 🔒 унаследован от родителя (только чтение) | ✅ редактируется |
| Апартамент внутри кондо (child) | 🔒 унаследован от родителя (только чтение) | ✅ редактируется |

**Ответственный агент:**

| Тип объекта | Поведение |
|---|---|
| Родительский резорт / кондо | Редактируется в Edit Panel. Изменение каскадируется на **все** дочерние объекты автоматически. |
| Дочерний дом / апартамент | 🔒 Унаследован от родителя. Ручное изменение заблокировано (пикер скрыт). |
| Отдельный дом (standalone) | Редактируется в Edit Panel. Каскада нет. |

> Новый дочерний объект (`create-unit`) автоматически наследует `responsible_agent_id` родителя в момент создания.

## 6.1 Review-flow (актуальный)

| Уведомление | Кнопка | Открывает | Действия |
|------------|--------|-----------|---------|
| `property_submitted` | 🔍 Посмотреть и принять решение | Правая панель (`WebPropertyEditPanel`, readOnly) | Одобрить / Отклонить |
| `edit_submitted` | 🔍 Посмотреть и принять решение | Правая панель (merged preview) | Одобрить / Отклонить |
| `property_approved` | — | — | Агент видит статус |
| `property_rejected` | — | — | Агент видит статус + история |

> **Важно:** `DiffModal` сохранён в коде, но **не используется** в review-flow. Обе проверки идут через правую панель.

## 6.2 Статусы объекта

| Статус | ⏳ в левой карточке | Правая панель (Admin) | Правая панель (Agent) |
|--------|--------------------|-----------------------|-----------------------|
| `pending` | ✅ | Approve / Reject block | "На проверке" badge (без Edit) |
| `approved` | ❌ | Стандартный вид | Стандартный вид |
| `rejected` | ✅ | "На обработке" + Edit + История | "На обработке" + Edit + История |

## 6.3 История отклонений

- Блок **"История отклонений"** показывается в `PropertyDetail` при `rejected` или при наличии legacy `rejection_reason`.
- Нумерованный список: #1 (первое) → #N (последнее). Последнее — жирное/красное, предыдущие — приглушённые.
- История хранится в таблице `property_rejection_history` (append-only).
- При auto-approve (admin edit + save) история **не удаляется**, блок скрывается только в `approved` состоянии.

## Phase 1: Унификация словаря ролей (2026-03-30)

> Совместимость: все существующие экраны продолжают работать без изменений.

### Новые поля профиля пользователя

| Поле | Тип | Пример |
|------|-----|--------|
| `user.plan` | string | `'standard'` / `'premium'` / `'korshun'` |
| `user.teamRole` | string\|null | `'agent'` / `'admin'` / `null` |
| `user.isAgentRole` | boolean | `true` если role=agent в company_members |
| `user.isAdminRole` | boolean | `true` если владелец компании |

### Правила гардов (рекомендация Phase 1 → обязательно Phase 2)

- Delete approved: `!isAgentRole` вместо `!isAgent` (применено в WebPropertiesScreen, PropertyDetailScreen).
- Остальные гарды (`canEditInfo`, `canBook` и т.д.) мигрируются позднее.
- Legacy: `isAgent = !!user?.teamMembership` продолжает работать (эквивалентно `isAgentRole`).
- Роль `worker` удалена: оставлены только `admin` и `agent`.
