## INC-001: Ошибка `property_rejection_history` not found in schema cache

**Дата:** 2026-03-28  
**Severity:** High (блокировало reject-flow)

**Симптом:**  
После применения миграции `20260328000000_property_rejection_history.sql` сервис при вставке в `property_rejection_history` возвращал ошибку: "relation property_rejection_history does not exist" или "not found in schema cache".

**Root cause:**  
Supabase кэширует схему БД. После применения миграции через CLI или Dashboard необходимо перезапустить клиент или дождаться обновления кэша. Либо миграция была применена не в нужной среде (local vs production).

**Fix:**  
1. Убедиться, что миграция применена в правой среде (`supabase db push` или через Dashboard).
2. Перезапустить Supabase Edge Functions / перезагрузить клиент.
3. Проверить через `to_regclass('public.property_rejection_history')`.

**Prevention:**  
- После каждой миграции запускать Sanity Check из `11_SQL_Sanity_Check_после_миграций.md`.
- Держать staging-среду синхронизированной с production.

## INC-002: Ошибка `properties.updated_at` column not found

**Дата:** 2026-03-28  
**Severity:** High (блокировало все reject-операции)

**Симптом:**  
При вызове `rejectProperty` или `rejectPropertyDraft` Supabase возвращал:  
`"Could not find the 'updated_at' column of 'properties' in the schema cache"`

**Root cause:**  
В реализации функций `reject` было добавлено `updated_at: new Date().toISOString()` в объект `.update()` для принудительного обновления записи и отслеживания времени. Колонка `updated_at` отсутствует в таблице `properties` (не была создана в базовой схеме).

**Fix:**  
1. Удалить `updated_at` из `.update()` в `rejectProperty` и `rejectPropertyDraft` в `src/services/propertiesService.js`.
2. Для принудительного обновления истории в UI ввести `historyRefreshKey` (ADR-002).

**Prevention:**  
- Перед добавлением нового поля в `.update()` убедиться, что колонка существует в таблице (проверить schema или миграции).
- Использовать TypeScript или задокументированный schema map для проверки полей.

## INC-003: История отклонений не обновлялась после второго reject при одинаковом статусе

**Дата:** 2026-03-28  
**Severity:** Medium (UX-проблема, история в БД корректна)

**Симптом:**  
После reject #2 (при уже `rejected` статусе) история на странице объекта не обновлялась. Пользователь видел только #1 без ручного reload.

**Root cause (часть 1):**  
`useEffect` в `PropertyDetail` имел deps `[property?.id, property?.property_status]`. При повторном reject из того же `rejected` состояния `property_status` не менялся → `useEffect` не перезапускался.

**Root cause (часть 2):**  
`rejectPropertyDraft` не обновлял `properties.property_status` (только draft). Второй reject через draft flow не менял основную запись объекта.

**Fix (часть 1):**  
Ввести `historyRefreshKey` в `WebPropertiesScreen`, инкрементировать в `onReject` и в `useEffect([refreshKey])`.

**Fix (часть 2):**  
В `rejectPropertyDraft` добавить явное `properties.update({ property_status: 'rejected', rejection_reason: reason })`.

**Prevention:**  
- При добавлении новых `useEffect` с deps на статус/причину — учитывать, что при повторных действиях значения могут не измениться.
- Использовать явные refresh-ключи для UI-инвалидации вместо poll-based deps.

## INC-004: Review-панель была невидима (два Modal одновременно)

**Дата:** 2026-03-23  
**Severity:** High (фича review недоступна)

**Симптом:**  
Кнопка "Посмотреть и принять решение" в уведомлении не открывала review-панель (панель невидима).

**Root cause:**  
`PropertyReviewModal` открывался как `<Modal>`, а основной popup уведомлений также был открытым `<Modal>`. Два Modal одновременно конфликтовали на мобильном/web renderer.

**Fix:**  
В `handleViewReview` перед `setReviewModal(...)` явно закрывать основной popup: `setOpen(false); openRef.current = false;`.

**Prevention:**  
- Не открывать несколько Modal одновременно.
- Предпочитать боковые панели (`Animated.View`) над `<Modal>` для review-flow.

## INC-005: Silent fail при approve/reject — UI обновлялся несмотря на ошибку

**Дата:** 2026-03-28  
**Severity:** Medium (некорректное состояние UI)

**Симптом:**  
При ошибке в `handleApprove` / `handleReject` (например, RLS rejection) уведомление помечалось как "action_taken" и исчезало из списка. Пользователь думал, что действие выполнено.

**Root cause:**  
`markActionTaken` и `setNotifs` вызывались вне `try...catch`, независимо от результата операции.

**Fix:**  
Перенести `markActionTaken` и `setNotifs` внутрь `try`. В `catch` показывать `window.alert(error.message)`.

**Prevention:**  
- State-обновление UI выполнять только после подтверждения успеха операции.
- Никогда не использовать `console.warn` как единственный способ сообщить об ошибке критического действия.
