---

## Обязательный цикл работы

```
Diagnose → Implement → Verify → Document
```

1. **Diagnose** — прочитать релевантные файлы перед правкой. Никогда не гадать по памяти.
2. **Implement** — минимальное, точечное изменение. Не переписывать то, что не сломано.
3. **Verify** — проверить линтер. Прогнать QA-сценарий если затронута бизнес-логика.
4. **Document** — обновить `08_Журнал_решений_ADR.md` и профильный файл.

---

## Правила коммуникации

- **Фиксировать решения в ADR** — любой архитектурный выбор (даже небольшой) фиксируется в `08_Журнал_решений_ADR.md`.
- **Не молчать об ошибках** — если операция упала, пользователь должен получить явный фидбек (alert/toast), не только `console.warn`.
- **Отчёт после задачи** — каждая задача завершается кратким отчётом: что изменено, почему, что не тронуто.

## Технические ограничения (всегда)

- ❌ Не добавлять `postgres_changes` подписки без явного запроса
- ❌ Не добавлять `window events` / `polling` для синхронизации
- ❌ Не трогать Flutter/mobile без явного запроса
- ✅ Использовать текущую архитектуру: push-signal → `refreshKey` → `load()`
- ✅ Проверять линтер после каждого изменения
