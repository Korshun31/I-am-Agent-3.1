## Обязательный цикл работы

```
Diagnose → Implement → Verify → Document
```

1. **Diagnose** — прочитать релевантные файлы перед правкой. Никогда не гадать по памяти.
2. **Implement** — минимальное, точечное изменение. Не переписывать то, что не сломано.
3. **Verify** — проверить линтер. Прогнать QA-сценарий если затронута бизнес-логика.
4. **Document** — обновить `08_Журнал_решений_ADR.md` и профильный файл.

## Правила коммуникации

- **Фиксировать решения в ADR** — любой архитектурный выбор (даже небольшой) фиксируется в `08_Журнал_решений_ADR.md`.
- **Не молчать об ошибках** — если операция упала, пользователь должен получить явный фидбек (alert/toast), не только `console.warn`.
- **Отчёт после задачи** — каждая задача завершается кратким отчётом: что изменено, почему, что не тронуто.

## Акцентные цвета и стиль UI

- Основной акцент: `#3D7D82` (teal)
- Фон акцента: `#EAF4F5`
- Danger (muted): `#FFF5F5` / `#FFCDD2` / `#C62828`
- Pending (amber): `#FFF8E1` / `#FFE082` / `#795548`
- Neutral bg: `#F4F6F9`, border: `#E9ECEF`, muted text: `#6C757D`

### UI parity rule (mobile tabs)

- Верхняя зона ключевых mobile-вкладок (`База`, `Booking calendar`, `Agent calendar`) должна использовать единый паттерн из 2 блоков:
  1) `header`: заголовок по центру + колокольчик справа;
  2) `toolbar`: строка поиска слева + action-иконки справа.
- Допустимые различия: текст заголовка, длина строки поиска, количество action-иконок во втором блоке.
- Недопустимые различия: размеры/позиции/отступы `title`, `bell`, `badge`, `search`, иконок тулбара между вкладками.
- Любая правка верхней зоны одной вкладки требует проверки визуальной консистентности со всеми вкладками этого паттерна.

## Зафиксированные бизнес-правила (LOCKED RULES)

> Правила ниже утверждены Human Owner и не могут быть изменены кодом или миграцией без явного подтверждения.  
> Если новая задача конфликтует с любым из этих правил — **остановиться и запросить подтверждение у Human Owner** перед реализацией.

---

### LOCK-001: Удаление объекта — снят 2026-04-30 (этап 2, ADR-015)

Старая логика «агент может удалить только не-approved объект» убрана вместе с модерацией. Новая логика: агент удаляет свой объект, если выполнены два условия:
1. Агент является ответственным: `properties.responsible_agent_id = auth.uid()`
2. У агента включена галочка `can_manage_property = true`

Роль `admin` — удаление любых объектов компании в рамках существующих RLS-политик.

### Role model checkpoint (2026-03-30)

- Действующие роли команды: только `admin` и `agent`.
- `worker` удалён из действующей модели ролей; исторические упоминания допустимы только как legacy-контекст.
- Термин `owner` в старых миграциях/записях трактуется как legacy-синоним текущей роли `admin`.
- SQL-подтверждение:
  - `company_members_role_check` → только `admin/agent`;
  - `company_members_status_check` → только `active/inactive`.

---

### Процессное правило: конфликт → спросить Human Owner

Если поступающая задача противоречит любому LOCKED RULE из этого раздела:  
1. Не реализовывать изменение.  
2. Явно сообщить о конфликте.  
3. Ждать подтверждения Human Owner перед любыми изменениями кода или SQL.

## Канонические поля профиля (Phase 1)

> Введено в Phase 1 (2026-03-30). Легаси-поля пока не удалены.

| Поле | Тип | Описание |
|------|-----|----------|
| `user.plan` | `'standard' \| 'premium' \| 'korshun'` | Тарифный план (billing) |
| `user.teamRole` | `'agent' \| 'admin' \| null` | Роль в команде компании |
| `user.isAgentRole` | `boolean` | `true` только для `role='agent'` |
| `user.isAdminRole` | `boolean` | `true` для владельца компании |

**Правило для нового кода:** использовать `isAgentRole`/`isAdminRole` вместо `!!teamMembership`.  
Легаси-поля `user.role`, `user.teamMembership`, `user.teamPermissions` остаются рабочими до Phase 2.

**Совместимость** (`company_members.role`):
- `'owner'` → мигрировано в `'admin'` (Phase 1 migration 20260330000002)
- `'agent'` → без изменений
- `'worker'` → мигрировано в `'admin'` (migration 20260330000003, роль удалена)
