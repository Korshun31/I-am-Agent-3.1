# 📊 Бизнес-сценарии и статусы объектов

---

## Жизненный цикл объекта

```
[Создание агентом]
     ↓
  pending  ←──────────────────────────────┐
     ↓                                    │
[Проверка админом]                        │
     ↓                                    │
 approved ──→ [Активный объект]           │
 rejected ──→ "На обработке" [→ Edit] ───┘
```

---

## Сценарий 1: Создание и первичная проверка

**Актор:** Agent  
**Действие:** Создаёт объект через `WebPropertyEditPanel` (mode='create')  
**Результат:** `properties.property_status = 'pending'`  
**Уведомление:** admin получает `property_submitted`  

**Admin видит:**
- Уведомление с кнопкой "🔍 Посмотреть и принять решение"
- Открывается `WebPropertyEditPanel` в `readOnly=true, reviewMode=true`
- Кнопки "Одобрить" / "Отклонить с причиной"

**UI-индикатор у обоих:** ⏳ чип в левой карточке + правой панели

---

## Сценарий 2: Отклонение

**Актор:** Admin  
**Действие:** Нажимает "Отклонить", вводит причину  
**Результат:**  
- `properties.property_status = 'rejected'`
- `properties.rejection_reason = reason` (последняя причина)
- Новая запись в `property_rejection_history`  
**Уведомление:** agent получает `property_rejected`

**UI у агента и админа:**
- ⏳ чип в левой карточке (сохраняется при `pending` ИЛИ `rejected`)
- В правой панели: чип "На обработке" + кнопка "Изменить"
- Блок "История отклонений" с нумерованным списком

---

## Сценарий 3: Повторная отправка и повторное отклонение

**Актор:** Agent  
**Действие:** Нажимает "Изменить", редактирует, отправляет черновик (`edit_submitted`)  
**Admin:** получает уведомление `edit_submitted`  
**Действие Admin:** открывает правую review-панель (не DiffModal!), отклоняет снова  
**Результат:**  
- `property_drafts.status = 'rejected'`
- `properties.property_status = 'rejected'` (явно обновляется)
- `properties.rejection_reason = новая причина`
- Новая запись в `property_rejection_history`

**UI:** история накапливается, последняя запись жирная/красная, предыдущие приглушены.

---

## Сценарий 4: Автопринятие при admin-edit rejected

**Актор:** Admin  
**Действие:** Открывает объект со статусом `rejected`, нажимает Edit, сохраняет  
**Триггер:** `WebPropertyEditPanel.handleSave` обнаруживает `isCompanyAdmin && property_status === 'rejected'`  
**Результат:**  
- `properties.property_status = 'approved'`
- `properties.rejection_reason = ''` (очищается)
- Обычный `updateProperty` с доп. полями  
**Уведомление:** agent получает `property_approved`

**UI:** объект становится обычным `approved`, ⏳ чип исчезает, блок истории скрывается (только в approved-состоянии).

> **Важно:** История в `property_rejection_history` **не удаляется** — остаётся как аудит-лог.

---

## Сценарий 5: Одобрение без правок

**Актор:** Admin  
**Действие:** Нажимает "Одобрить" в review-панели  
**Результат:**  
- `properties.property_status = 'approved'`
- `properties.rejection_reason` не меняется  
**Уведомление:** agent получает `property_approved`

---

## Статусы и их UI-значение

| `property_status` | ⏳ в карточке | Правая панель (admin) | Правая панель (agent) |
|-------------------|--------------|----------------------|----------------------|
| `pending` | ✅ | Approve/Reject block | "На проверке" badge |
| `approved` | ❌ | — | — |
| `rejected` | ✅ | "На обработке" + Edit | "На обработке" + Edit |

---

## Review-панель: edit_submitted vs property_submitted

| Тип уведомления | Источник данных | Открывает |
|----------------|-----------------|-----------|
| `property_submitted` | `properties` напрямую | `WebPropertyEditPanel` (readOnly) |
| `edit_submitted` | `{ ...property, ...(draft.draft_data || {}) }` — merged preview | `WebPropertyEditPanel` (readOnly) |

Оба открываются в **правой боковой панели**. `DiffModal` больше не используется для review.
