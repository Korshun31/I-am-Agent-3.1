# 📊 Бизнес-сценарии и статусы объектов

---

## Жизненный цикл объекта

```
[Создание агентом или админом]
     ↓
[Активный объект]  ─→  [Редактирование]  ─→  [Удаление]
```

С 2026-04-30 (этап 2 — упрощение прав) объект сразу становится активным. Никаких промежуточных статусов «на проверке» / «отклонён» — ни админ, ни агент через них не проходят.

---

## Сценарий 1: Создание объекта

**Актор:** Admin или Agent (при `can_manage_property = true`)  
**Действие:** Создаёт объект через `WebPropertyEditPanel` (mode='create') или мобильный визард  
**Результат:**  
- `properties.property_status = 'approved'` через DB-default (колонка остаётся до этапа 3, не используется)  
- Объект публикуется и виден в списке сразу  
**Уведомление:** если создатель — агент, админу приходит `property_created` (информационная плашка «агент добавил объект»). Никакого review-flow. Если создаёт админ — уведомления нет.

## Сценарии 2-5 — сняты 2026-04-30 (этап 2)

Сценарии «Отклонение», «Повторная отправка», «Автопринятие при admin-edit rejected», «Одобрение без правок» — больше неактуальны. Модерация выпилена:
- Статусы `pending` / `rejected` не используются.
- Уведомления `property_submitted`, `property_approved`, `property_rejected`, `edit_submitted`, `edit_approved`, `edit_rejected` удалены из кода и БД.
- Кнопки «Одобрить / Отклонить» убраны из UI.
- Таблицы `property_drafts` и `property_rejection_history` физически остаются до этапа 3 (cleanup-миграция дропнет их), но не пишутся и не читаются.
- Функции `approveProperty`, `approvePropertyDraft`, `rejectProperty`, `rejectPropertyDraft`, `submitPropertyDraft` удалены из `propertiesService`.

См. RLS-миграцию `20260429000001_simplify_properties_rls_phase2.sql` и коммиты `0a6371e`, `e7cf7b3`, `d81efa3`, `63950ce`.

## Сценарий 6: Переназначение ответственного

**Актор:** Admin  
**Действие:** Открывает Edit Panel объекта, меняет «Ответственный» на другого агента (или на Компанию / null)  
**Результат:**  
- `properties.responsible_agent_id` обновляется  
- Предыдущий агент **немедленно теряет** SELECT-доступ к объекту (RLS срабатывает при следующем запросе)  
- Новый агент получает доступ

**Для дочерних объектов (дома в резорте / апартаменты в кондо):**  
Ответственный на дочернем объекте не редактируется вручную. Меняется только на родителе.

---

## Сценарий 7: Каскад ответственного на дочерние объекты

**Актор:** Admin  
**Действие:** Меняет ответственного на родительском резорте или кондо  
**Механизм:**  
1. Edit Panel сохраняет изменения родителя через `updateProperty`  
2. Сразу вызывается `updatePropertyResponsible(parentId, agentId, cascade=true)`  
3. Supabase обновляет `responsible_agent_id` у всех записей с `resort_id = parentId` одним запросом

**Результат:**  
- Все дочерние дома / апартаменты получают нового ответственного атомарно  
- Если агент был удалён из ответственных (null), он теряет доступ ко всем объектам внутри немедленно  
- Новый агент получает доступ к родителю + всем дочерним

**Правило для дочерних объектов:**  
- Пикер ответственного на странице редактирования дочернего объекта **скрыт** (App и Web)  
- В Edit Panel дочернего объекта отображается метка «Унаследован от родителя 🔒»  
- Попытка отправить `responsible_agent_id` из дочернего wizard-редактора исключена на уровне `buildUpdates`

## Ролевой checkpoint (2026-03-30)

- Для всех сценариев в этом документе действуют только роли `admin` и `agent`.
- Упоминания `owner`/`worker` в старых заметках считаются legacy-контекстом:
  - `owner` в исторических записях соответствует текущей роли `admin`;
  - `worker` не является действующей ролью.
- SQL-подтверждение модели:
  - `company_members_role_check` = `admin/agent`;
  - `company_members_status_check` = `active/inactive`.
