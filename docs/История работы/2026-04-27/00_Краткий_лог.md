## Важные решения дня
- **24 часа vs 7 дней для magic-link**: Supabase Pro имеет потолок Email OTP expiration 86400 сек. Разделили: приглашение в `company_invitations` живёт 7 дней (наша БД), magic-link 24 часа. При истечении ссылки — кнопка «Отправить повторно».

### Orphan-проблема (нашли архитектурный косяк)
- Korshun: «вчера приглашал `gedorekiv@gmail.com`, сегодня отозвал, не могу пригласить заново — пишет “email already registered”».
- Корень: триггер `handle_new_user` создавал `users_profile` СРАЗУ при `inviteUserByEmail` (до того как агент ткнёт в письмо). После revoke в users_profile оставалась orphan-запись, email считался занятым.
- Усугубилось: я не выписал табличку жизненного цикла перед Этапом 4 — записал правило в memory `feedback_entity_lifecycle_check.md` чтобы в будущем не пропускать.
- Решение (3 миграции + правки кода):
  - `20260427000003`: `handle_new_user` при invite_token → RETURN NEW (не создаёт ни профиль, ни компанию). Профиль создаётся только в `WebInviteAcceptScreen.handleSubmit` (UPSERT после агентского submit формы).
  - `20260427000004`: служебная RPC `get_auth_user_id_by_email` для Edge Function (поиск orphan по email).
  - `20260427000005`: уникальный частичный индекс на `(company_id, lower(email)) WHERE status IN ('sent','pending')` — против двойного клика admin'а.
  - Edge Function: при `orphan` → если последнее приглашение sent/pending+active → отказ `EMAIL_OCCUPIED_ORPHAN_PENDING` (с подсказкой «используй Resend»). Иначе → `admin.deleteUser(orphan)` → recheck (race protection) → обычный invite.
  - `WebInviteAcceptScreen.handleSubmit`: INSERT users_profile, на конфликт по id — UPDATE только name (не затирать role существующего юзера).
  - i18n: добавлены ключи `EMAIL_OCCUPIED_ORPHAN_PENDING`, `EMAIL_RACE`, `inviteSessionInvalidTitle/Message` на en/th/ru.
- Cleanup: однократно удалили `gedorekiv@gmail.com` из `users_profile` + `auth.users` чтобы переключиться на новый flow.
