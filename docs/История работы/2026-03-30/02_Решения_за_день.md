## Принятые решения

### Решение 1: Доступ агента к объектам — только по `responsible_agent_id`

| Поле | Значение |
|------|----------|
| **Decision** | Убрать `user_id`-ветку из RLS и сервисного фильтра; оставить только `responsible_agent_id = auth.uid()` |
| **Why** | Агент-создатель сохранял доступ после переназначения ответственного — нарушение Company-First; данные компании должны контролироваться admin через ответственность, а не фактом создания |
| **Impact** | RLS migration + `propertiesService.getProperties` + документация |
| **Owner** | Human (бизнес-правило) + Agent-Developer (реализация) |
| **Связанный ADR** | ADR-010 |

### Решение 2: Каскад ответственного при смене на родительском резорте/кондо

| Поле | Значение |

### Решение 4: Дочерний объект не может изменить ответственного вручную

| Поле | Значение |
|------|----------|
| **Decision** | Для `isChildUnit` / `isHouseInResort` пикер ответственного скрыт полностью; `buildUpdates` не включает `responsible_agent_id`; Edit Panel показывает readonly-метку |

| **Impact** | `WebPropertyEditPanel.js`, `PropertyEditWizard.js` |

### Решение 5: LOCK-001 — Зафиксировано правило удаления объектов по роли

| Поле | Значение |
|------|----------|
| **Decision** | Ограничение на удаление `approved` объектов применяется **только** к роли `agent`. Роли `worker` и `admin` — не ограничены этим правилом. Агент может удалить объект только если он создатель (`user_id`) и статус не `approved`. |
| **Why** | Было некорректно задокументировано как ограничение для «всех участников компании» — это слишком широко. Правило относится только к role=`agent`. |

| **Связанный ADR** | ADR-010, LOCK-001 |

## Финальные закрытия дня (checkpoint)

### Блок A: Web bugfix — `Invalid Date` + регресс клика

- **Проблема:** в Gantt-календаре Web появлялся `Invalid Date`; после первичного фикса клик по свободной ячейке перестал запускать создание.
- **Причина:** невалидная координата клика и зависимость только от `locationX` на web.
- **Решение:** сохранены strict-guards дат, добавлен безопасный резолвер X-координаты с fallback.
- **Проверка:** `Invalid Date` не появляется; клик по свободной ячейке и кнопка `+ Add booking` работают корректно.
- **Итог:** обе web-ошибки закрыты.

### Блок B: Mobile bugfix — месяц календаря из `BookingCalendarScreen`

- **Проблема:** при создании брони из `BookingCalendarScreen` календарь открывался с начала ленты (январь 2025), а не с актуального месяца.
- **Причина:** расчёт `monthKey` по локальному X без учёта текущего горизонтального scroll.
- **Решение:** расчёт переведён на абсолютную координату внутри таймлайна; при невалидном индексе передаётся `null`.
- **Проверка:** в текущем и проскролленном месяце открывается корректный шаг 2 календаря; path из `PropertyDetailScreen` без изменений.
- **Итог:** баг воспроизведения начала ленты устранён.

### Блок C: Role model checkpoint — только `admin/agent`

- **Проблема:** legacy-термины `owner/worker` в документах могли читаться как актуальные правила.
- **Причина:** исторические артефакты миграций и старых записей.
- **Решение:** в уставе и ADR явно закреплено: действующие роли только `admin/agent`; `owner`/`worker` отмечены как legacy.
- **Проверка:** зафиксированы SQL-constraint подтверждения:
  - `company_members_role_check` = `admin/agent`;
  - `company_members_status_check` = `active/inactive`.
- **Итог:** единая актуальная ролевая модель документально закреплена.

### Блок D: 3-я вкладка — Mobile Booking calendar UI parity

- **Проблема:** 3-я вкладка (`Booking calendar`, mobile) визуально отличалась от паттерна `База` в верхней зоне экрана.
- **Причина:** шапка и тулбар были реализованы с иными размерами/позициями элементов, из-за чего при свайпе между вкладками возникал заметный визуальный разрыв.
- **Решение:** зафиксирован единый паттерн верхней зоны:
  - блок 1: `title center + bell right`;
  - блок 2: `search + filter`;
  - календарная сетка начинается ниже второго блока;
  - поиск по `name/code` фильтрует строки календаря локально.
- **Проверка:** визуальная консистентность между `База` и `Booking calendar` достигнута; поиск отрабатывает по названию и коду; функционал календаря без регрессий.
- **Итог:** доработка 3-й вкладки завершена, изменение зафиксировано как UI-only без изменения бизнес-логики.

### Блок E: P0.x stabilisation — единый контракт типов и owner-комиссий

- **Проблема:** после внедрения новых unit-типов в отдельных потоках сохранялись остаточные несовпадения App/Web: не везде учитывались `resort_house/condo_apartment`, в agent ping-pong часть rejected-resubmit шла мимо draft-модерации, а owner-комиссии хранили лишние `is_from` флаги.
- **Причина:** фрагментарная миграция legacy-веток (`house`) и устаревших условий (`needsApproval` по `can_edit_info` без статуса `rejected`), плюс незавершённая очистка старых owner-commission атрибутов.
- **Решение:** зафиксирован house-like dual-read в ключевых App/Web экранах, role-save для unit в App унифицирован (`agent -> pending`, `admin -> approved`), для rejected edit в Web принудительно используется `submitPropertyDraft` с `edit_submitted`; owner-комиссии в App wizard доведены до схемы `value + is_percent` с UI-переключателем `฿/%`; legacy `owner_commission_*_is_from` удалены из Web-кода и вынесены в отдельную SQL-миграцию удаления колонок.
- **Проверка:** lint без ошибок; hotfix `useCallback` на Web применён; create type race в App закрыт; `rg` по `src` не находит `owner_commission_one_time_is_from` и `owner_commission_monthly_is_from`; базовые smoke-сценарии create/edit и moderation проходят.
- **Итог:** parity между App и Web по критическим create/moderation/pricing сценариям восстановлен, контракт owner-комиссий упрощён и формализован.
