## TD-133 — переписать storage `property-photos` под бизнес-правило

**Правило владельца**: доступ к фото = доступ к объекту.
- Админ компании-владельца объекта — всегда.
- Активный ответственный агент этого объекта — пока он назначен. Сняли с объекта — потерял доступ.
- Создатель файла не имеет значения. У агента нет «своего» — всё компании.

**Текущая дыра**: на бакете `property-photos` (PUBLIC) политики INSERT/UPDATE/DELETE открыты любому `authenticated` без условия. Любой залогиненный юзер другой компании теоретически может через DevTools стереть/подменить фотки твоих объектов.

**Препятствие реализации**: в имени файла нет связи `file → property`. Текущая структура путей в `src/services/storageService.js`:
- Фотки объектов: `<user_id>/<timestamp>_<random>.jpg` (плюс `_thumb.jpg`).
- Аватары/логотипы: `avatars/<user_id>/avatar_<timestamp>.<ext>`.

Связь file ↔ property хранится только в массиве URL внутри `properties.photos` (JSONB). Из storage policy туда не прыгнуть.

**Согласованный план** (на отдельную сессию ~полдня-день):
1. Ввести явную таблицу `property_photos(id, property_id, storage_path, is_thumb, ...)` ИЛИ включить `property_id` в путь файла.
2. Переписать `src/services/storageService.js`:
   - `uploadPhoto(localUri, propertyId)` — принимает property_id явно;
   - `uploadPhotoWithThumb(localUri, propertyId)` — то же для оригинала+миниатюры;
   - `deletePhotoFromStorage(url)` — продолжает работать через парсинг URL.
3. Бэкфилл-скрипт: пройтись по всем `properties` и для каждой фотки в `photos[]` либо сделать запись в `property_photos`, либо переместить файл в новый путь.
4. Storage policy на `storage.objects` — SECURITY DEFINER функция-чекер `can_modify_property_photo(name TEXT) RETURNS BOOLEAN` с проверкой: caller — admin компании этого property ИЛИ caller = `properties.responsible_agent_id`.
5. Аватары/логотипы — отдельным правилом «только владелец трогает свои».

Подвох планирования: фотки загружаются ДО создания property — в визарде юзер сначала добавляет фото, потом сохраняет объект. Тогда `property_id` ещё нет. Нужно либо двухфазный flow (черновик property → загрузка фото), либо временный bucket-prefix для драфтов.

**Где описан полный план**: `docs/PROGRESS_PLAN.md`, TD-133 ⬜.

## Что важно НЕ путать

- `bookings.commission` — комиссия ОТ клиента (% или сумма от стоимости аренды). Это видят все, кто видит бронь.
- `bookings.owner_commission_one_time / _monthly / monthly_breakdown` — что риелтор получает ОТ собственника объекта. **Это НЕ секрет владельца агентства** (как сначала подумал security-аудитор). Агент сам вводит эту цифру при оформлении брони — он должен её видеть в своей же брони.
