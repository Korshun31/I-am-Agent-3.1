# 📋 Краткий лог — pre-launch security audit (2026-05-15 — 2026-05-16)

## Фокус сессии
Перед публичным запуском CRM прогнали полный security-аудит prod-окружения. Запустили встроенный `security-review` (только текущая ветка — он ничего не нашёл, потому что на ветке `main` пусто), затем подняли отдельного `general-purpose` агента-параноика с задачей пройтись по всему prod: RLS-политики на всех таблицах, тела SECURITY DEFINER функций, edge-функция `invite-agent`, конфиг auth, storage. Аудитор вернул отчёт с 12 находками: 3 критические, 4 средние, 5 низких. Закрыли 9 из 12 одной коммитной серией `4f766ef`. Одна критическая отложена в TD-133 (политики storage на фотки) — требует рефакторинга как хранятся файлы. Две «дыры» сняты как «не баг» после уточнения бизнес-логики с владельцем.

## Архитектурные проверки

Перед каждой деструктивной операцией (DROP функций / политик / FK CASCADE / правка edge-функции) запускался `Plan` subagent ("архитектор-параноик") с конкретным брифом — найти все callers, проверить cross-references из тел других PL/pgSQL функций, обнаружить VIEW / триггеры / RLS-политики с зависимостями, оценить риск регрессии. Все вердикты были «безопасно», правки соответствовали отчётам.

Архитектор по TD-140 нашёл ещё два «забытых» orphan-канала (`locations`, `calendar_events`), которые в первоначальном плане TD-140 не упоминались — закрыли в той же миграции одним пакетом.

## Что НЕ сделано — намеренно

| Из отчёта | Решение |
|---|---|
| **TD-133 (critical)** — storage policies на `property-photos` | Отложено в отдельную сессию. Сейчас политики INSERT/UPDATE/DELETE открыты любому `authenticated` (бакет PUBLIC, 4 политики в Dashboard: upload/select/update/delete). Бизнес-правило владельца: «доступ к фото = доступ к объекту» (админ компании всегда, ответственный агент пока он назначен). Препятствие: в имени файла нет связи file→property — нужна явная таблица `property_photos(id, property_id, storage_path, ...)` или включение `property_id` в путь, переписать `src/services/storageService.js` (`uploadPhoto` / `uploadPhotoWithThumb` / `deletePhotoFromStorage`), бэкфилл-скрипт по существующим фоткам. На это нужно полдня-день. Договорились держать как acceptable risk до отдельной сессии. |
| **fail-open в `checkMembershipActive`** (medium из отчёта) | Снято как «не записываем». RLS + ужесточённые SELECT-политики из TD-128 уже отрезают уволенного от всех данных компании на уровне БД. Клиентский fail-open даёт ему только пустой UI-shell без полезных данных. Селективная блокировка одного RPC на iOS/Android без джейлбрейка фактически невозможна. Цена «правильного» retry+timeout фикса — заметная (новая абстракция, риск ложных выкидываний в метро/самолёте). Не блокер запуска. |
| **column-level RLS на `bookings.owner_commission_*`** (low из отчёта) | Снято как «не баг, фича». Архитектор-аудитор думал, что комиссия собственника — секрет владельца агентства, но Korshun уточнил: агент сам её вводит при оформлении брони, значит должен видеть в своей же брони. Нормальный workflow. |
