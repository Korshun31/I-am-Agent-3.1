## Принятые решения

### Решение 1: `historyRefreshKey` вместо `properties.updated_at`

| Поле | Значение |
|------|----------|
| **Decision** | Ввести local state `historyRefreshKey` для инвалидации истории отклонений |
| **Why** | Колонка `updated_at` отсутствует в `properties`; добавление колонки требует миграции и рисков |
| **Impact** | Только `WebPropertiesScreen.js` — no schema changes, no new subscriptions |
| **Owner** | Agent-Developer |
| **Связанный ADR** | ADR-002, ADR-003 |

---

### Решение 2: `historyRefreshKey` инкрементируется в `useEffect([refreshKey])`

| Поле | Значение |
|------|----------|
| **Decision** | Любой глобальный refreshKey-сигнал (включая из Bell) инкрементирует historyRefreshKey |
| **Why** | Bell-flow использует broadcast → refreshKey, не локальный onReject; без этого история не обновлялась |
| **Impact** | Единый путь для всех reject-источников |
| **Owner** | Agent-Developer |
| **Связанный ADR** | ADR-003 |

---

### Решение 3: Создать `docs/Устав компании/` как отдельный контур

| Поле | Значение |
|------|----------|
| **Decision** | 12 файлов базы знаний в `docs/Устав компании/` |
| **Why** | Множество принятых решений нигде не зафиксированы; новый агент не имеет контекста |
| **Impact** | Только документация, runtime не затронут |
| **Owner** | Agent-Docs |
| **Связанный ADR** | Все ADR-001..ADR-008 |

## Отклонённые варианты

| Вариант | Почему отклонён | Кто отклонил |
|---------|----------------|--------------|
| Добавить `updated_at` в `properties` миграцией | Риск: затрагивает все UPDATE-запросы, потребует изменений в сервисах; нецелесообразно ради одного refresh-триггера | Human |
| Realtime подписка на `property_rejection_history` | Противоречит архитектурному принципу (только targeted refresh) | Agent-Developer |
| `window.dispatchEvent` как постоянный механизм | Создаёт второй путь обновления, труднее дебажить | Agent-Architect |
