Общие правила

I am Agent — Кодекс v2.0
Эти правила действуют во ВСЕХ модулях системы. Они определяют как работает управление доступом, как обновляются данные, как добавляются переводы, как показываются ошибки, как проверяется ввод.

1. Управление доступом (RBAC)

Роли и разрешения

✅
У каждого пользователя есть роль. Для каждой роли определены два слоя:
  • Жёсткие правила — зашиты в систему, никто не может изменить
  • Настраиваемые разрешения — переключатели, которые настраивает тот кто пригласил
При добавлении новой роли — задаются оба слоя. Приложение не переписывается.

Текущие роли

Админ — зарегистрировался сам, владелец компании. Может всё.

Агент — приглашён админом. Настраиваемые переключатели:

ПереключательЕсли выключен
Добавлять объектыКнопка «+» скрыта
Редактировать информациюИзменения через черновик
Редактировать ценыИзменения цен через черновик
БронироватьКнопка бронирования скрыта
Удалять бронированияКнопка удаления скрыта

Жёсткие правила агента (нельзя изменить): не видит ответственного, не может одобрять/отклонять, не может удалить одобренный объект, видит только свои данные, не может управлять командой.

Обычный пользователь (без команды) — как админ, но без команды.

Планируемые роли: Owner (независимый собственник), Owner (приглашённый собственник).

Главный принцип

✅
В приложении проверяется не роль («ты агент?»), а разрешение («тебе можно добавлять объекты?»). Не важно какая роль — важно что конкретно разрешено.

Двойная защита

✅
Первая стена — база данных сама не отдаёт чужие данные. Вторая стена — приложение дополнительно проверяет. Две стены лучше одной.

Невидимость

✅
Если пользователь не может выполнить действие — кнопка не показывается вообще. Не серая, не заблокированная — полностью невидимая.

2. Работа с данными

✅
Не копировать данные. Если данные уже есть в общем хранилище — использовать оттуда. Копия устаревает, оригинал обновляется — пользователь видит старое.
✅
После изменения — обновить через хранилище. Не подставлять данные вручную. Пусть система загрузит свежие и экран обновится автоматически.
✅
Не использовать устаревшие данные. Если подождал ответ от сервера — данные в переменной могли устареть. Бери свежие из хранилища.

3. Переводы

✅
Каждая новая строка текста — во все три языка одновременно (en, ru, th). Нельзя добавить только на одном.
✅
Перед добавлением — поискать в файле переводов. Может такая строка уже есть. Не создавать дубли.

4. Обновление в реальном времени

✅
После любого изменения данных — обновить у себя + отправить сигнал другим участникам. Это происходит автоматически внутри сервисов.
✅
Сигнал — лёгкий (как рация). Не передаёт данные, только говорит «обновись». Сервер не нагружается.

5. Ошибки

✅
Ошибки показываются на экране (красный текст), не во всплывающем окне. Исключение: подтверждение удаления.
✅
Текст ошибки — на языке пользователя. Не технический текст из базы данных.

6. Проверка данных

✅
Обязательные поля проверяются в приложении перед отправкой. Пользователь не ждёт ответ сервера чтобы узнать что забыл заполнить имя.
✅
Критичные ограничения дублируются в базе данных. Если проверка в приложении сломается — база не пропустит плохие данные.
✅
Проверки одинаковые на вебе и мобильном. Нельзя: на вебе три обязательных поля, а на мобильном одно.
